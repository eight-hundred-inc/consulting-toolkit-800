# 報告書: Agent Team テンプレート

統合分析の方向性が固まった段階で、報告書ドラフトの作成と多角レビューを並行で実行する。

## 前提条件

- `Output/提案書.md` が存在し、最終的な論点構造が確定していること
- `Output/報告書骨子.md` が存在すること
- `Output/インタビューまとめ.md` が存在すること
- `Output/調査/` にデスクリサーチ結果が格納されていること
- 分析の方向性についてユーザーと合意が取れていること

## チーム構成

- **Lead（リードエージェント）**: 統合分析と報告書ドラフトを担当
- **Fact Checker**: ドラフト完了後、事実・数字・出典を検証
- **Devil's Advocate**: ドラフト完了後、論理の飛躍・前提の疑義・反対意見を提示

## team-prompt パターン

```
最終報告書の作成とレビューを Agent Team で実行してください。

## 入力
- 報告書骨子: Output/報告書骨子.md
- インタビューまとめ: Output/インタビューまとめ.md
- 調査結果: Output/調査/ 配下の全ファイル
- 提案書: Output/提案書.md
- 分析結果: Output/分析結果.md（あれば）

## ワークフロー

### Phase 1: ドラフト作成（Lead 単独）
1. 報告書骨子に沿って、インタビューまとめ・調査結果・分析結果を統合する
2. 章ごとに主張・根拠・示唆を構成する
3. Output/最終報告書.md にドラフトを出力する

### Phase 2: 並列レビュー（ドラフト完了後）
ドラフトが完成したら、以下のチームメイトを spawn してレビューを並列実行する。

- Fact Checker: Output/最終報告書.md を読み、以下を検証する。
  1. 引用されている数字・統計の正確性
  2. 出典の存在と信頼性
  3. 事実関係の矛盾がないか
  4. 調査結果やインタビュー内容の引用が正確か
  quality-reviewer SubAgent の品質チェック手法を参考にすること。
  検証結果をメッセージで Lead に送ること。

- Devil's Advocate: Output/最終報告書.md を読み、以下の観点から批判的レビューを行う。
  1. 論理の飛躍や根拠が弱い主張はないか
  2. 暗黙の前提に疑義はないか
  3. 反対意見や代替シナリオは検討されているか
  4. 結論が提案書の論点に対して十分に答えているか
  レビュー結果をメッセージで Lead に送ること。

### Phase 3: 統合（Lead）
1. Fact Checker と Devil's Advocate のレビュー結果を受け取る
2. 指摘事項を報告書に反映する
3. 修正版を Output/最終報告書.md に上書きする
4. 全成果物のパス、レビューで反映した指摘事項の要約を提示し、ユーザーのレビューを待つ

## ルール
- Phase 1 が完了するまで Fact Checker と Devil's Advocate は spawn しない
- レビュー結果はファイルではなくメッセージで Lead に送る
- 最終的なファイル編集は Lead だけが行う
```

## 成果物

| 成果物 | 格納先 | 担当 |
|--------|--------|------|
| 最終報告書 | Output/最終報告書.md | Lead |

## カスタマイズのポイント

- 分析結果（Output/分析結果.md）が未作成の場合は、Lead が統合分析から実行する
- プロジェクト種類に応じてレビュー観点を調整する:
  - 技術動向調査: 技術的正確性、最新性の検証を強化
  - 市場調査: 市場規模の算出根拠、競合分析の網羅性を重視
  - デューデリジェンス: リスク評価の漏れ、財務データの整合性を重点チェック
- 報告書をスライド形式にする場合は、Phase 3 の後に 800-branded-pptx スキルでの変換を追加する
