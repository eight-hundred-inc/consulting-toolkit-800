# 各層の書き方ガイド

## 層1: 全体結論（エグゼクティブサマリ）

**役割**: 文書全体のメッセージを凝縮する。読み手がここだけ読んでも調査の主要な結論が把握できる。

### SCR（Situation-Complication-Resolution）で導入する

全体結論の冒頭では、いきなり論点番号を並べるのではなく、読み手が「なぜこの調査が必要だったか」を瞬時に理解できるフレーミングを置く。

- **Situation（状況）**: 読み手が既に知っている前提・背景を1-2文で共有する
- **Complication（複雑化要因）**: その状況において何が問題・課題・不確実性になっているかを1-2文で述べる
- **Resolution（解決の方向性）**: この調査が導いた結論の核心を1文で述べる

SCR の後に、大論点ごとの結論を配置する。SCR は読み手を結論に引き込むための「滑走路」であり、長くなりすぎないようにする（合計3-5文）。

### 構成要素

1. **大論点ごとの端的な結論**（各2-3文）
   - 番号付けは本文の大論点と1対1で対応させる
   - 大論点①の結論が全体結論の①、大論点②が②...という対応関係を崩さない
   - 全体結論に独自の切り口や番号体系を持ち込まない
   - 結論で使う分類・枠組みの名称（例: 「3類型」「2軸」等）は、本文を先読みしなくても意味が取れるよう1行で補足する
   - 結論文には必ず主語を入れる（「貴社にとって〜」「○○市場では〜」）。主語の省略により、市場全体のトレンドなのかクライアント固有の示唆なのかが判別できなくなることを防ぐ

2. **横断的示唆**（1段落）
   - 個別の大論点結論を並べるだけでなく、論点を横断して統合するとどのような示唆が導けるかを述べる
   - 「①で〜が判明し、②で〜が確認されたことから、③の方向性として〜が整合する」のように因果の接続を示す
   - 大論点の中に「統合的な結論」「戦略オプション」等の総括セクションが既にある場合は、その存在を明示して参照する形でもよい

### サイズ制約

- 層1の目標は文書全体の5-8%以内とする。500行の文書なら25-40行が目安
- 大論点が4つ以上ある場合、各結論を1-2文に圧縮し、層1が肥大化しないよう制御する
- 層1で根拠を展開しない（SKILL.md Step 5 の反復排除ルールと連動）。根拠は「→第X章参照」で委ねる
- 層1が目標を超過した場合、根拠の展開が混入していないか、横断的示唆が冗長でないかを確認する

## 層2: 大論点サマリ

**役割**: 各大論点の見出し直後に置き、配下の小論点すべてを統合した上位の結論を示す。

### 書き方

- 大論点の見出し直後、最初の小論点に入る前に記載する
- 2-4文で、その大論点全体を通じて得られた結論を述べる
- 小論点の内容を列挙・要約するのではなく、小論点を統合して得られる上位メッセージを書く
- 読み手がサマリだけ読めば大論点の結論がわかり、小論点はその根拠・詳細として位置づけられる構造にする
- **枠組みの導入**: 結論が類型・分類・フレームワークに依存する場合、結論文の前に1-2行で枠組みを提示する。読者はまだ下位の小論点を読んでいないため、結論を受け取るための最低限のボキャブラリーをここで渡す
- **用語の初出定義**: サマリ内で初めて登場する専門用語・略語は、括弧書きで簡潔な説明を添える
- **主語の明示**: 結論文（特に矢印結論 → ...）には必ず主語を入れる。「○○型への移行が収益構造上の自然なパスとなる」ではなく「貴社にとって○○型への移行が〜」と書く

### 大論点間のトランジション

前の大論点の結論を受けて次の大論点に接続する「つなぎ」を、大論点サマリの冒頭に1文で入れる。

- 「大論点①で市場構造を確認したが、次に問題となるのは〜」のように前の結論を踏まえて進む
- 最初の大論点にはトランジションは不要（SCR が導入の役割を果たす）
- トランジションにより、文書全体が「ブツ切りの分析レポート」ではなく「一つの思考の流れ」として読めるようになる

### ダミー例

```markdown
## 大論点① 競合他社の事業モデルと収益構造

対象市場のプレーヤーは、自社で運用まで担う**自社運用型**、SW基盤を
ライセンス提供する**プラットフォーム型**、両者を組み合わせた
**ハイブリッド型**の3類型に分かれる。課金はサブスクリプション型が
主流だが、ライセンス単体の収益は小さく、導入支援・運用代行を含む
サービス提供が実質的な収益ドライバーとなっている。貴社が位置する
プラットフォーム型は収益規模に構造的な限界があり、貴社にとって
運用サービスの要素を加えたハイブリッド型への段階的移行が、
収益構造上の自然なパスとなる。

### 小論点①-1: 主要な収益源はどこにあるか

（ここから層3の記述が始まる）
```

## 層3: 小論点の分析

**役割**: 個別論点ごとの結論と、それを支えるファクト・データを一体的に記述する。

### 構成パターン

```markdown
### 小論点X-Y: 問いの形式で書く

結論を1-2文で端的に述べる。

根拠となるファクトを展開する。具体的なデータや事例を挙げ、
結論と根拠の因果関係が伝わるように書く（情報源ID）。

| データ表がある場合 | ここに配置する |
| --- | --- |
| 項目A | 値A |
| 項目B | 値B |

追加のファクトや、貴社にとっての含意を述べる。
```

### 根拠の構造化

根拠が3つ以上ある場合、フラットな箇条書きにすると各根拠の論理的役割が見えなくなる。各根拠が結論をどの角度から支えているかを、接続語またはラベルで明示する。

**典型的な論理構造パターン**:
- なぜ必要か（現状の制約・限界）→ 実現可能か（参入障壁・移行条件）→ なぜこのクライアントか（固有の優位性・差別化要因）
- 市場側の根拠（需要・規制・トレンド）→ 技術側の根拠（成熟度・差別化）→ 経済性の根拠（コスト・収益構造）
- 現状の制約 → 打開の条件 → 推奨アクション

```markdown
### 小論点①-4: 貴社の参入優位性はどこにあるか

貴社はハイブリッド型への移行において、SW・HW（ハードウェア）双方を
自社で保有する点で市場上の希少な参入優位性を持つ。

- **移行の必要性**: 市場の課金構造はベネフィットシェア型（運用収益の
  10-15%を報酬として受け取るモデル）が主流であり、ライセンス収入
  単体では収益規模に構造的な限界がある（①-1）
- **移行の実現性**: 契約設計は免責ベースが業界標準であり参入障壁は
  高くない。一方、制御精度のデータ蓄積が実質的な差別化要因となる
  （①-2, ①-3）
- **貴社の固有優位**: 主要14社の技術領域を比較した結果、SW領域の
  コモディティ化が確認された。HWを自社製造する貴社の特性は、
  SW＋HW統合制御という参入障壁の高い領域で優位に立てる（①-4調査）
```

### 小論点間の用語依存

前の小論点で導入した概念・略語が次の小論点で登場する場合も、初出扱いで簡潔に補足する。読者が小論点を飛ばし読みする可能性があるためである。補足は括弧書き1つ程度で十分であり、冗長な再説明は不要。

### 「含意」の書き方

含意は「So What? への答え」であり、小論点の分析を大論点の結論に接続するための最も重要なパラグラフである。

**良い含意の条件**:
- この調査・分析固有の発見から導かれた、読み手にとって新しい示唆を含む
- 読み手の意思決定や行動に直結する具体性がある（「何をすべきか」「何を避けるべきか」が見える）
- 上位の大論点サマリのメッセージを構成する「部品」として機能する

**避けるべき含意**:
- ✕ 「今後の動向を注視する必要がある」（一般論で行動を示唆しない）
- ✕ 「XXは重要な要素である」（重要だと言うだけで何も伝えていない）
- ✕ 「慎重な検討が求められる」（検討の方向性が不明）
- ○ 「貴社が運用レイヤーに参入するなら、A社の価格帯（XX万円/拠点）を下回る設計が必須となる。ただしXX以下では収益性を確保できないため、対象は年間XX拠点以上の大口顧客に限定すべきである」

### ダミー例

```markdown
### 小論点①-1: 主要な収益源はどこにあるか

ライセンス収入は年間約200万円/拠点にとどまり、収益の柱は運用代行
にある（B01）。運用代行は拠点数の増加に対して限界コストが低下する
スケーラブルな構造を持つ。

A社の費用体系が具体的に判明した。

| 項目 | 金額 |
| --- | --- |
| システム利用料（基本） | XXX万円/年 |
| 運用サービス（基本） | XXX万円/年 |
| 業務委託費 | XXX万円/箇所 |

貴社がプラットフォーム提供のみで収益を確保するのは現時点で困難であり、
何らかの形で運用レイヤーへの関与が必要となる。
```

### 表の活用指針

- 費用体系・比較・制度変更一覧等は表で視認性を確保する
- 単一ソースのファクト表（上記例のような費用体系表）に加え、複数ソースの横断比較表（例: 競合3社を同一観点で並べて比較する表）も有効
- 表の前後に、表が示すポイントを文章で補足する（表だけで結論を語らない）

## 総括大論点の扱い

大論点構造の中に「戦略オプション」「統合的な結論」等の総括的な大論点がある場合、その大論点は他の大論点の結論を横断的に統合する役割を持つ。この場合:

- 全体結論の横断的示唆は、この総括大論点の結論を要約・参照する形で書く
- 総括大論点のサマリには、結論文を冒頭に置く（「XX方向を優先し、YY方向は既存市場の深耕に集中する」等）

### 戦略オプションを含む場合の必須記述項目

戦略オプション（A/B/C/D等の選択肢）を提示する場合、各オプションに以下の3要素を必ず含める:

1. **(a) 具体像とトレードオフ**: 何をするのか、何を得て何を失うのか
2. **(b) 必要な体制・リソース・パートナーシップ**: 実行に必要な人材・組織体制・外部パートナーを明示的に切り出す。トレードオフの記述に埋没させない
3. **(c) 一次情報に基づく評価**: インタビュー等で得られた実務者の評価・見解。見解が分かれる場合は対立見解として両論を提示する

(b)はトレードオフ記述の中に「運用人材の確保が必要」等として埋もれやすい。独立した項目として切り出し、読み手が各オプションの実行条件を一覧で比較できるようにする。

## 情報源の参照ルール

### 参照形式

- **単独参照**: `（A01）`、`（B01）` -- 特定の情報源を根拠として示す
- **複合参照**: `（A01＋デスクトップ調査）` -- 複数の異なるソースが同一の結論を支えている場合に使用し、根拠の厚みを示す

### 情報源IDの体系

プロジェクトのインタビュー議事録・デスクリサーチレポートで使用されているID体系をそのまま踏襲する。

### 仮説ラベルの判断基準

排除すべきもの（作業プロセスのラベル）:
- 「仮説: ○○」「検証結果: 支持/棄却」
- 「修正が必要」「部分的に支持」
- 「提案書での仮説に対し〜」

許容されるもの（結論の確度に関する留保）:
- 「本分析は○○を前提としており、適用にあたっては前提条件の読み替えが必要」
- 「○○については調査対象が限定的であり、追加検証を推奨する」
- 「○社の事例から抽象化した枠組みであり、一般化には留意が必要」

判断基準: **作業の経過を報告しているか、それとも結論の信頼性について読み手に有用な情報を提供しているか**。前者は排除、後者は許容する。

## 文章レベルの品質基準

一流コンサルティングファームの報告書に求められる文章品質を満たすためのガイドライン。

### 一文一義

- 一つの文に複数のメッセージを詰め込まない
- 「〜であり、〜だが、〜ため、〜である。」のような長い複文を避け、短い文に分割する
- 目安: 1文は60-80字以内。100字を超える文は分割を検討する

### 主語と述語の対応

- 主語と述語の距離を近づける。修飾語句が間に入りすぎると論旨がぼやける
- 「〜は、…（長い修飾）…、…（さらに修飾）…、○○である。」→ 修飾部分を別文にする

### 対比・比較で切れ味を出す

ファクトを並べるだけでなく、対比構造を活用してメッセージを鋭くする。

- 「A社はXX方式を採用している」→「A社はXX方式を採用する一方、B社はYY方式を選択しており、この違いは○○に起因する」
- 対比は「違い」だけでなく「なぜ違うのか」「その違いは読み手にとって何を意味するか」まで踏み込む

### 能動態を基本とする

- ✕ 「市場の拡大が見込まれている」（受動態・主体不明）
- ○ 「XX調査機関は市場が年率X%で拡大すると予測している」（主体明確）
- ✕ 「導入が推奨される」
- ○ 「貴社はXXの導入を優先すべきである」

### 抽象語の空転を避ける

以下のような表現は、具体的な内容に置き換えるか、削除する:

- 「最適化」→ 何をどう変えるのか具体的に書く
- 「シナジー」→ どの事業とどの事業が、何をどう共有・連携するのか
- 「DX推進」→ どの業務プロセスを、どの技術で、どう変えるのか
- 「戦略的に重要」→ なぜ重要なのか、何に対して重要なのかを書けば「戦略的に」は不要になる
