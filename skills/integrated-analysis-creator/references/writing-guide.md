# 各層の書き方ガイド

## 層1: 全体結論（エグゼクティブサマリ）

**役割**: 文書全体のメッセージを凝縮する。読み手がここだけ読んでも調査の主要な結論が把握できる。

### 構成要素

1. **大論点ごとの端的な結論**（各2-3文）
   - 番号付けは本文の大論点と1対1で対応させる
   - 大論点①の結論が全体結論の①、大論点②が②...という対応関係を崩さない
   - 全体結論に独自の切り口や番号体系を持ち込まない

2. **横断的示唆**（1段落）
   - 個別の大論点結論を並べるだけでなく、論点を横断して統合するとどのような示唆が導けるかを述べる
   - 「①で〜が判明し、②で〜が確認されたことから、③の方向性として〜が整合する」のように因果の接続を示す
   - 大論点の中に「統合的な結論」「戦略オプション」等の総括セクションが既にある場合は、その存在を明示して参照する形でもよい

---

## 層2: 大論点サマリ

**役割**: 各大論点の見出し直後に置き、配下の小論点すべてを統合した上位の結論を示す。

### 書き方

- 大論点の見出し直後、最初の小論点に入る前に記載する
- 2-4文で、その大論点全体を通じて得られた結論を述べる
- 小論点の内容を列挙・要約するのではなく、小論点を統合して得られる上位メッセージを書く
- 読み手がサマリだけ読めば大論点の結論がわかり、小論点はその根拠・詳細として位置づけられる構造にする

### ダミー例

```markdown
## 大論点① 競合他社の事業モデルと収益構造

対象市場ではサブスクリプション型の課金が主流だが、ライセンス単体の
収益は小さく、導入支援・運用代行を含むサービス提供が実質的な収益
ドライバーとなっている。プレーヤーは自社運用型・プラットフォーム型・
ハイブリッド型の3類型に分かれ、貴社が位置するプラットフォーム型は
収益規模に限界がある。運用サービスの要素を加えたハイブリッド型への
移行が、収益構造上の自然なパスとなる。

### 小論点①-1: 主要な収益源はどこにあるか

（ここから層3の記述が始まる）
```

## 層3: 小論点の分析

**役割**: 個別論点ごとの結論と、それを支えるファクト・データを一体的に記述する。

### 構成パターン

```markdown
### 小論点X-Y: 問いの形式で書く

結論を1-2文で端的に述べる。

根拠となるファクトを展開する。具体的なデータや事例を挙げ、
結論と根拠の因果関係が伝わるように書く（情報源ID）。

| データ表がある場合 | ここに配置する |
| --- | --- |
| 項目A | 値A |
| 項目B | 値B |

追加のファクトや、貴社にとっての含意を述べる。
```

### ダミー例

```markdown
### 小論点①-1: 主要な収益源はどこにあるか

ライセンス収入は年間約200万円/拠点にとどまり、収益の柱は運用代行
にある（B01）。運用代行は拠点数の増加に対して限界コストが低下する
スケーラブルな構造を持つ。

A社の費用体系が具体的に判明した。

| 項目 | 金額 |
| --- | --- |
| システム利用料（基本） | XXX万円/年 |
| 運用サービス（基本） | XXX万円/年 |
| 業務委託費 | XXX万円/箇所 |

貴社がプラットフォーム提供のみで収益を確保するのは現時点で困難であり、
何らかの形で運用レイヤーへの関与が必要となる。
```

### 表の活用指針

- 費用体系・比較・制度変更一覧等は表で視認性を確保する
- 単一ソースのファクト表（上記例のような費用体系表）に加え、複数ソースの横断比較表（例: 競合3社を同一観点で並べて比較する表）も有効
- 表の前後に、表が示すポイントを文章で補足する（表だけで結論を語らない）

## 総括大論点の扱い

大論点構造の中に「戦略オプション」「統合的な結論」等の総括的な大論点がある場合、その大論点は他の大論点の結論を横断的に統合する役割を持つ。この場合:

- 全体結論の横断的示唆は、この総括大論点の結論を要約・参照する形で書く
- 総括大論点のサマリには、結論文を冒頭に置く（「XX方向を優先し、YY方向は既存市場の深耕に集中する」等）

## 情報源の参照ルール

### 参照形式

- **単独参照**: `（A01）`、`（B01）` -- 特定の情報源を根拠として示す
- **複合参照**: `（A01＋デスクトップ調査）` -- 複数の異なるソースが同一の結論を支えている場合に使用し、根拠の厚みを示す

### 情報源IDの体系

プロジェクトのインタビュー議事録・デスクリサーチレポートで使用されているID体系をそのまま踏襲する。

### 仮説ラベルの判断基準

排除すべきもの（作業プロセスのラベル）:
- 「仮説: ○○」「検証結果: 支持/棄却」
- 「修正が必要」「部分的に支持」
- 「提案書での仮説に対し〜」

許容されるもの（結論の確度に関する留保）:
- 「本分析は○○を前提としており、適用にあたっては前提条件の読み替えが必要」
- 「○○については調査対象が限定的であり、追加検証を推奨する」
- 「○社の事例から抽象化した枠組みであり、一般化には留意が必要」

判断基準: **作業の経過を報告しているか、それとも結論の信頼性について読み手に有用な情報を提供しているか**。前者は排除、後者は許容する。
