# 統合分析結果作成スキル

デスクリサーチ・インタビュー等の調査結果を統合し、論点構造に沿った分析結果文書を作成する。

## 入出力

- **入力（必須）**: 提案書（論点・小論点の構造定義）、デスクリサーチレポート（1つ以上）
- **入力（任意）**: インタビュー議事録（0件以上）、プロジェクトの CLAUDE.md
- **出力**: `Output/分析結果.md`

## ワークフロー

**このスキルを使うときは必ず** `references/writing-guide.md` を読み、各層の書き方に従うこと。

### Step 1: 入力ファイルの確認

1. 提案書から大論点・小論点の構造を抽出する
2. デスクリサーチレポート、インタビュー議事録の一覧を確認する
3. CLAUDE.md があれば文書作成ガイドライン・スコープを確認する

### Step 2: 論点構造の確定

1. 提案書の大論点・小論点をそのまま分析結果の骨格として採用する
2. 調査過程で新たに判明した論点がある場合は、既存構造に統合する（勝手に論点体系を再構成しない）
3. 小論点の見出しは問い形式を維持する

### Step 3: 小論点ごとの分析（層3の作成）

各小論点について、以下の順序で記述する:

1. **結論**（1-2文）: その小論点に対する端的な答えを冒頭に置く
2. **根拠**: 結論を支えるファクト・データ・事例を展開する
3. **表**: 費用体系・競合比較・制度一覧等がある場合は表で視認性を確保する
4. **含意**: 読み手（クライアント）にとっての意味・示唆を述べる

情報源の参照（A01、B01等）は根拠の信頼度を示すために記載する。

### Step 4: 大論点サマリ・全体結論の作成（層2・層1の作成）

**ボトムアップで作成する**（小論点 → 大論点サマリ → 全体結論の順）:

1. 各大論点について、配下の小論点すべてを統合した上位メッセージを2-4文で書く（層2）
2. 全体結論として、大論点ごとの端的な結論（各2-3文）を番号付きで書く（層1）
3. 横断的示唆を1段落で書く（層1）

### Step 5: 品質チェック

本スキル末尾の品質チェックリストに基づき、全項目を確認する。違反があれば修正する。

## 文書構成: 3層のピラミッド

分析結果は以下の3層で構成する。上の層ほど抽象度が高く、下の層が根拠・詳細を担う。

```
┌──────────────────────────────────────────────┐
│  層1: 全体結論（エグゼクティブサマリ）          │  ← 1回のみ
│    ① 大論点①の結論（2-3文）                    │
│    ② 大論点②の結論（2-3文）                    │
│    ③ ...大論点の数だけ繰り返し                  │
│    横断的示唆（1段落）                          │
├──────────────────────────────────────────────┤
│  層2: 大論点①サマリ                            │  ← 大論点の数だけ繰り返す
│  ┌────────────────────────────────────────┐   │
│  │  層3: 小論点①-1 の分析                  │   │  ← 小論点の数だけ繰り返す
│  ├────────────────────────────────────────┤   │
│  │  層3: 小論点①-2 の分析                  │   │
│  └────────────────────────────────────────┘   │
├──────────────────────────────────────────────┤
│  層2: 大論点②サマリ                            │
│  ┌────────────────────────────────────────┐   │
│  │  層3: 小論点②-1 の分析                  │   │
│  ├────────────────────────────────────────┤   │
│  │  層3: 小論点②-2 の分析                  │   │
│  └────────────────────────────────────────┘   │
├──────────────────────────────────────────────┤
│  層2: 大論点③サマリ                            │
│  │  ...同様に繰り返し                          │
└──────────────────────────────────────────────┘
```

読み手はどの層から読んでも必要な粒度の情報を得られる設計とする。

## 4つの原則

### 原則1: 結論を先に述べ、根拠で支える

各レベル（全体結論・大論点サマリ・小論点）において、結論を冒頭に置き、それを支えるファクト・データ・表を続ける。積み上げ型（根拠→結論）ではなく、トップダウン型（結論→根拠）で書く。

### 原則2: 仮説検証の作業プロセスを残さない

アウトプットには結論とその根拠のみを記載する。以下の要素は文書に含めない:

- 「**仮説**: ...」「**検証結果**: ...」等のラベル
- 「修正が必要」「部分修正」「支持」「部分的に否定」等の検証ステータス
- 「提案書で設計した仮説では〜」等の作業プロセスへの言及
- 「キックオフMTGで〜氏が指摘」等の社内イベントへの言及

ただし、結論の確度に関する留保（「本分析は○○を前提としており、適用にあたっては前提条件の読み替えが必要」等）は、結論の一部として許容する。

### 原則3: 読み手はクライアントである

文書の読み手はクライアント自身であることを常に意識する:

- 「クライアント」「先方」「お客様」→ 社名または「貴社」
- 自社の作業プロセス（提案書の仮説設計、キックオフでの議論等）への言及は削除
- 冒頭説明文も「仮説の検証結果と結論を示す」等ではなく「分析結果と示唆を示す」とする

### 原則4: ソース別列挙ではなく論点別統合

デスクリサーチの知見とインタビューの知見をソース単位で列挙しない。各小論点の中で、出所の異なる情報を結論と根拠の関係で一体的に記述する。

- ✕ 「デスクリサーチからの知見: ...」「インタビューからの知見: ...」と分けて列挙
- ○ 結論を述べ、その根拠としてデスクリサーチ・インタビュー双方のファクトを織り交ぜて記述

インタビュー対象者間で見解が分かれた場合は、対立見解として整理し、それぞれの論拠を明示する。

## 品質チェックリスト

文書完成後、以下を確認する:

- [ ] 全体結論の番号付けが本文の大論点と1対1で対応しているか
- [ ] 全体結論に横断的示唆（論点間の統合メッセージ）が含まれているか
- [ ] 各大論点の見出し直後にサマリパラグラフがあるか
- [ ] 「仮説」「検証結果」等の作業プロセスラベルが残っていないか
- [ ] 「修正が必要」「支持」等の検証ステータスが残っていないか
- [ ] 「クライアント」「先方」「お客様」等の社内目線表現が残っていないか
- [ ] 冒頭説明文が結論中心の記述になっているか
- [ ] 各小論点が結論→根拠の順で書かれているか
- [ ] ソース別の知見列挙（「デスクリサーチからの知見」「インタビューからの知見」）ではなく、論点ごとに統合された記述になっているか
