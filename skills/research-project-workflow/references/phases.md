# フェーズ・ステップ詳細

各ステップの詳細な実行手順。

## 共通手順: desk-researcher SubAgent の起動

Step 1（内部）と Step 9 で共通して使用する。差分（モード・入力ファイル）は各ステップの実行手順で指定する。

```
1. ユーザーに以下を確認:
   - 入力ファイルのパス（各ステップで異なる）
   - 出力先フォルダのパス
   - Browser Use で参照したいサイト（ある場合）
2. Task ツールで desk-researcher SubAgent を起動する（subagent_type="desk-researcher"）。
   prompt には各ステップで指定されたコンテキストファイルの絶対パス、出力先フォルダの絶対パス、モード、追加指示を含める
3. SubAgent が内部で desk-research Skill を読み込み、調査を実行する（親エージェントは Skill を直接実行しない）
4. SubAgent が返却した結果を受け取る:
   - 調査レポートのファイルパス
   - 主要な発見事項
   - 仮説検証の要約（hypothesis-driven モードの場合）
```

## Phase 0: 提案

### Step 1: 論点・仮説の設計 [AI]

**review_level**: full

**目的**: 与件から調査→論点構造化→仮説検証までを一貫して実行し、検証済みの論点・仮説を設計する

**入力**:
- `Input/与件.md` または与件情報

**出力**:
- `Output/論点・仮説.md`（検証済み）
- 調査レポート（ユーザー指定の出力先フォルダ）
- `Output/プロジェクトサマリ.md`（更新）

**実行手順**:

PM が以下の4フェーズを段階的に実行する。

#### フェーズ1: 前提整理

与件から調査の方向性を構造化する。

1. `Input/与件.md` を読み込む
2. 以下の要素を整理する:
   - **クライアントの背景・状況**: 企業名、業界、現在の課題
   - **調査の目的**: 何の意思決定に使うか（投資判断、参入判断、戦略策定など）
   - **調査スコープ**: 対象領域・除外領域
   - **初期の調査方向性**: 論点の手前の「調べるべき方向」のリスト（3-7項目）
   - **既知情報と不明情報**: 与件で既に分かっていることと、まだ分かっていないこと
3. 整理結果をメモとして保持（ファイル出力は不要。次フェーズの入力として使用）

#### フェーズ2: 探索的調査

desk-researcher SubAgent を `exploratory` モードで起動し、広く事実を収集する。

1. 「共通手順: desk-researcher SubAgent の起動」に従い、以下のパラメータで実行する:
   - コンテキスト: フェーズ1で整理した前提（背景・目的・スコープ・調査方向性）をテキストで渡す。与件ファイルの絶対パスも含める
   - モード: `exploratory`
   - 出力先フォルダ: ユーザーに確認
2. SubAgent の返却結果（調査レポートのパス、主要な発見事項）を受け取る

#### フェーズ3: 論点・仮説の構造化

探索的調査の結果を踏まえ、論点と仮説を構造化する。

1. 調査レポートを読み込む
2. 調査結果から以下を設計する:
   - **大論点**（3-5個）: 問い形式で記載。事業設計に直結させる
   - **小論点**（各大論点に2-4個）: 問い形式で記載
   - **仮説**: 各論点に紐づく仮説。調査で得た数字・固有名詞を含め、根拠付きで記述する
   - **論点間の優先順位**: 意思決定への影響度で順位付け
3. 構造化した論点・仮説を作業用ドキュメントとして保持

#### フェーズ4: 仮説検証調査

desk-researcher SubAgent を `hypothesis-driven` モードで起動し、仮説を検証する。

1. 「共通手順: desk-researcher SubAgent の起動」に従い、以下のパラメータで実行する:
   - コンテキスト: フェーズ3で構造化した論点・仮説をテキストで渡す。与件ファイルの絶対パスも含める
   - モード: `hypothesis-driven`
   - 出力先フォルダ: フェーズ2と同じフォルダ
2. SubAgent の返却結果（調査レポートのパス、仮説検証の要約）を受け取る
3. 仮説検証シートの結果を論点・仮説に反映する:
   - **支持された仮説**: 根拠・ソースを追加して補強
   - **否定された仮説**: 仮説を修正または棄却、理由を記載
   - **新たな発見**: 新しい仮説を追加

#### 最終出力

1. `Output/論点・仮説.md` を生成する。以下の構成で記載:

```markdown
# 論点・仮説（検証済み）

## 調査概要
[前提整理の要約と調査の概要]

## 大論点①：[問い形式]

### 仮説
[検証結果を反映した仮説文。数字・固有名詞・根拠付き]

### 小論点①-1：[問い形式]
**仮説**: [検証結果を反映した仮説文]
**根拠**: [調査で得た根拠]
**検証状態**: 支持/否定/修正/未検証

### 小論点①-2：[問い形式]
...

## 大論点②：[問い形式]
...

## 論点間の優先順位
[意思決定への影響度による順位付けと理由]

## 未検証事項
[インタビュー等で補完が必要な項目]
```

2. `Output/プロジェクトサマリ.md` を更新:
   - 「調査の背景・目的」セクションを充実させる
   - 論点・仮説を抽出して書き込む。検証状態を含めて記載
   - 「検証の経過」に探索的調査・仮説検証調査で判明した内容を記述
   - 「現時点での示唆と提供価値」に初回記述を行う

**品質チェック**:
- 大論点が3-5個、問い形式で記載されているか
- 各大論点に2-4個の小論点が紐づいているか
- 仮説に数字・固有名詞・根拠が含まれているか
- 仮説の検証状態（支持/否定/修正/未検証）が全て記載されているか
- 未検証事項にインタビュー等での補完方法が明記されているか
- プロジェクトサマリの論点・仮説が同期されているか

**品質チェック**:

### Step 4: インタビューガイド作成 [AI]

**review_level**: full

**目的**: 提案書と調査結果をもとに、インタビューで検証・補強すべき事項を整理し、インタビューガイドを作成

**入力**:
- `Output/提案書.md`（論点・仮説）
- 調査レポート（Step 1 で生成）

**出力**:
- `Output/インタビューガイド.md`

**実行手順**:
1. `skills/interview-guide-creator/SKILL.md` を読み込む
2. 提案書から論点（大論点・小論点）と初期仮説を抽出
3. 調査レポートから検証済み事項と未検証事項を整理
4. スキルの指示に従ってインタビューガイドを作成

**スキルが出力する内容**:
- インタビューの目的と調査全体における位置づけ
- 検証すべき事項（論点との対応）
- 想定インタビュー対象者（タイプ別の属性・選定理由）
- インタビュー設問（対象者タイプ別、時間配分付き）

**品質チェック**:
- 全ての大論点に対応する質問セクションがあるか
- 各質問の「検証したいこと」が明記されているか
- 調査で不足していた情報を埋める質問があるか
- 回答から次のアクションに繋げられる設計か

### Step 9: デスクリサーチ [AI → SubAgent]

**review_level**: light

**目的**: インタビュー結果を踏まえ、情報が不足している領域の深掘り調査を実施する

**担当**: AI（`desk-researcher` SubAgent経由で実行。中間出力のコンテキスト分離が目的）

**実行手順**: 「共通手順: desk-researcher SubAgent の起動」に従い、以下のパラメータで実行する。

- コンテキストファイル: 提案書、インタビューまとめ、既存調査結果
- モード: `gap-filling`
- 追加指示: インタビューで判明したギャップを埋める深掘り調査

**品質チェック**:
- 調査レポートが出力先フォルダに格納されている
- インタビューで判明したギャップが埋められているか、未解決事項が明記されている
- 仮説検証シートが更新されている
- プロジェクトサマリが以下の通り更新されている:
  - 仮説テーブルの状態と検証の経過が調査結果と同期されている
  - 「現時点での示唆と提供価値」がデスクリサーチの結果を踏まえて更新されている
  - 発見事項ログに主要なファクトを追記している

## Phase 2: 分析・とりまとめ

### Step 10: 最終報告書作成 [AI] → integrated-analysis-creator

**review_level**: full

**目的**: デスクリサーチとインタビュー結果を統合分析し、最終報告書を作成する

**入力**:
- `Output/プロジェクトサマリ.md`（論点・仮説の検証状況・検証の経過・示唆と提供価値・リスク・主要発見事項）
- `Output/提案書.md`（論点・小論点の構造定義）
- `Output/報告書骨子.md`（章立て・構成定義）
- `Output/インタビューまとめ.md`
- デスクリサーチ結果

**出力**:
- `Output/最終報告書.md`
- `Output/プロジェクトサマリ.md`（最終更新: 検証の経過の最終化、示唆と提供価値の最終版記述、リスクの整理）

**実行手順**:
```
1. skills/integrated-analysis-creator/SKILL.md を読み込む
2. skills/integrated-analysis-creator/references/writing-guide.md も読み込む
3. Output/提案書.md から論点・小論点構造を抽出
4. Output/報告書骨子.md から章立て・構成定義を確認
5. Output/インタビューまとめ.md、デスクリサーチ結果を入力として渡す
6. スキルの指示に従って Output/最終報告書.md を作成
7. Output/プロジェクトサマリ.md を最終更新
```

**構成（3層ピラミッド）**:
```markdown
# 最終報告書

## 全体結論
① 大論点①の結論（1-2文）
② 大論点②の結論（1-2文）
③ ...
横断的示唆（1段落）

## 大論点①: [論点名]

サマリパラグラフ（2-4文: 配下の小論点を統合した上位結論）

### 小論点①-1: [問い形式]
結論（1-2文）→ 根拠（ファクト・データ・表）→ 含意

### 小論点①-2: [問い形式]
結論 → 根拠 → 含意

## 大論点②: [論点名]
サマリパラグラフ
### 小論点②-1: ...
### 小論点②-2: ...

...（大論点の数だけ繰り返し）

## 付録
用語集、参考文献一覧、データソース詳細等
```

**プロジェクトサマリの最終更新**:
最終報告書作成の完了後、`Output/プロジェクトサマリ.md` を最終版として整備する:
- 全仮説の最終判定を反映し、検証の経過を最終化する
- 「現時点での示唆と提供価値」をプロジェクト全体の結論として記述する（暫定ではなく確定版）
- リスク・懸念事項を整理し、残存リスクを明記する
- 「未解決の問いと今後の調査方針」を整理する（解決済みの問いは削除、残存する問いは今後の対応方針を記載）

**品質チェック**:
- 全体結論の番号付けが本文の大論点と1対1で対応しているか
- 全体結論に横断的示唆（論点間の統合メッセージ）が含まれているか
- 各大論点の見出し直後にサマリパラグラフがあるか
- 「仮説」「検証結果」等の作業プロセスラベルが残っていないか
- 「クライアント」「先方」「お客様」等の社内目線表現が残っていないか
- 各小論点が結論→根拠の順で書かれているか
- ソース別の知見列挙ではなく、論点ごとに統合された記述になっているか
- 論点に対する回答が明確か
- 根拠（データ・インタビュー）が示されているか
- アクショナブルな提言があるか
- 報告書骨子で定義された章立て・想定アウトプットが網羅されているか
- プロジェクトサマリが最終版として整備されているか（検証の経過、示唆と提供価値、リスク、未解決の問い）
