# Interview Minutes Creator

インタビューの質問リストと文字起こしから、構造化された詳細な議事録を作成する。

## ワークフロー

1. **入力ファイルの確認**
   - 質問リスト（質問項目と意図が記載されたファイル）
   - インタビュー文字起こし（音声認識等で生成されたテキスト）
   - クライアント文脈（提案書・調査設計書等）がある場合はその論点・仮説も把握する

2. **内容の分析**
   - 質問リストの各質問と対応する回答を文字起こしから特定
   - インタビュイーのプロフィール情報を抽出
   - 重要なファクト（数値、固有名詞、具体例）を抽出
   - 発言の背景・理由・判断根拠・比較対象・業界の空気感等、ファクト以外の情報も抽出

3. **議事録の構成**
   - 出力フォーマットに従って議事録を作成
   - 各質問への回答を第一階層（核心文）と第二階層（詳細）に整理
   - 文字起こしの口語表現を書き言葉（文語体）に変換する。証言の内容そのものを文の主語にし、報告フレーム（「〇〇氏は〜と述べた」等）は原則使わない
   - クライアント文脈（提案書・調査設計書）がある場合、まとめにおいて論点・仮説との対応関係を明示する

4. **品質チェック**
   - 全ての質問に回答が紐づいているか確認
   - 専門用語に説明が付与されているか確認
   - 第一階層の文章で内容の全体像が把握できるか確認
   - ファクトサマリが本文の全セクションの示唆を漏れなく集約しているか確認
   - 用語（同一概念を指す表現）が文書内で統一されているか確認
   - 同じ数字が異なる文脈で登場する場合、区別が明記されているか確認

## 出力フォーマット

詳細なフォーマットガイドは `references/output-format.md` を参照。

### 基本構成

```markdown
# [インタビュータイトル]

## インタビュー概要
- 実施日時：YYYY年MM月DD日（曜日）HH:MM〜HH:MM
- 所要時間：約XX分
- 実施形式：オンライン/対面

## インタビュイープロフィール
[氏名、所属、役割、専門性]

## ファクトサマリ
### [カテゴリ名]
[本文全セクションの示唆を漏れなく集約した箇条書き]

## インタビュー内容
### [セクション名]

**質問：[質問文をそのまま記載]**

- **[核となる情報を端的に表現した完全な文]**
    - [詳細1]
    - [詳細2]
    - ...

## まとめ
[クライアント文脈と対応づけた示唆の総括]
```

### デフォルトで含めないもの

- **逐語録**：ユーザーから要望があった場合のみ追加する
- **セパレーター（`---`）**：デフォルトでは使用しない

## 見出し階層ルール

見出しは `###` までとし、それより深い階層は使用しない。

- `#`：文書タイトル
- `##`：大セクション（インタビュー概要、プロフィール、ファクトサマリ、インタビュー内容、まとめ等）
- `###`：セクション見出し（セクション1：〇〇、等）およびファクトサマリのカテゴリ見出し

質問見出しは見出し記法（`#`）ではなく **太字テキスト** で表現する。

```markdown
### セクション1：〇〇

**質問：〜〜〜か**

- **核心文**
    - 詳細

**サブ質問：〜〜〜か**

- **核心文**
    - 詳細
```

## 記述ルール

### 第一階層（太字の核心文）
- 質問への回答を1-2文で端的に表現
- 必ず完結した文章で、単文で意味が理解できること
- 太字（`**`）で強調

### 第二階層（箇条書きの詳細）
- 第一階層を補足する具体的な情報
- 体言止めは極力使わない
- 以下の情報を文字起こしから可能な限り拾い上げて盛り込む
    - 数値（金額、期間、人数等）
    - 具体例（企業名、製品名、技術名等）
    - 判断の理由・背景（なぜそう考えるか、なぜそうなったか）
    - 比較・対比（AとBの違い、過去と現在の変化等）
    - インタビュイー自身の経験に基づく実感・体感
    - 業界の空気感・暗黙知

### 文体ルール

**主語の原則：**
- 回答セクションでは、証言の内容そのものを文の主語にする。インタビュー議事録である以上、回答セクションの発言者は自明であり、「誰が言ったか」を毎回フレーミングする必要はない
- 「〇〇氏は〜と述べた/強調した/指摘した/助言した/見解を示した」等の報告フレームは原則使わない
- インタビュイーの氏名が主語になる構文は、デスクリサーチとの矛盾点の明示など、発言者の立場・経歴が論旨に不可欠な場合に限って使用する

**語彙の原則：**
- 全体を書き言葉（文語体）で統一する。文字起こしの口語表現はそのまま転記せず、内容を書き言葉に変換する
- 口語的語彙は文書に適した表現に置き換える（例：お客さん→顧客、営業マン→営業担当者、全然→大きく、かなり→相当/大幅に、やる→実施する/行う）
- 「」の使用は固有名詞・制度名・検索ワード等に限定する。発言内容の引用には使わない

### 用語の扱い
- 専門用語は初出時に（）内で簡潔に説明
- 略語は初出時にフルスペルと説明を記載
- 業界特有の概念は具体例を交えて説明

### 品質基準
- 質問文は原文のまま使用
- インタビュー内容を可能な限り盛り込む（上記「第二階層」の情報粒度を参照）
- 単文で意味が理解できる完結した文章
- 重要なポイントは太字で強調

## ファクトサマリの設計指針

- ファクトサマリだけ読む読者がいる前提で設計する（社内会議で上層部が斜め読みするケース等）
- 本文の全セクションから示唆を集約し、本文にしか書かれていない重要情報がないようにする
- 同じ数字が異なる文脈で登場する場合（例：臨床試験5億円 vs 承認全体10億円）、ファクトサマリ内で区別を明記する

## まとめの設計指針

- 単なるインタビュー内容の要約ではなく、クライアントの意思決定に資する示唆として構成する
- クライアントの検討文脈（提案書・調査設計書の論点・仮説）が入力にある場合、それとの対応関係を明示する
- インタビュイーの見解がネガティブだった領域についても、どの条件なら成立するかという前向きな論点整理を行う
