# workflow.md テンプレート（活動カタログ型）

月次継続型プロジェクトの初期化時に、以下のテンプレートを使用して `workflow.md` を作成する。

## 適用条件

以下の特徴を持つプロジェクトに適用する。

- 期限・最終ゴールが固定されない月次継続の業務委託・顧問契約である
- 支援内容が「順序性のない活動の集合」であり、ステップの順次進行になじまない
- 定例MTG・月次レポート等の月次サイクルが存在する
- 新規施策・改善提案が随時発生する

明確な納期と最終成果物を持つ単発案件には適用しない（調査型または他類型を使う）。

## テンプレート

プレースホルダ（`[...]` と `{...}`）をヒアリング結果・与件で置換して使う。活動カテゴリ（B, C, ...)は支援内容に応じて必要な数だけ設ける。

```markdown
# プロジェクト: [プロジェクト名]

## 基本情報
プロジェクト概要・クライアント・期間等は CLAUDE.md を参照。
- **ワークフロー**: 活動カタログ型（カスタム）
- **ワークフローソース**: custom: workflow.md
- **形態**: 月次継続型の業務委託（期限・最終ゴールは固定されない）
- **管理方針**: 進捗ステップ型ではなく、活動カタログ＋履歴ログで管理する
- **開始日**: [YYYY-MM-DD]

## ファイル配置
- CLAUDE.md: プロジェクト概要・作業上の注意・文書作成ガイドライン（静的）
- workflow.md: 活動カタログ・進行中の活動・履歴ログ・意思決定記録
- Output/プロジェクトサマリ.md: 支援状況サマリ・課題・施策進捗・リスク・発見事項
- Input/: クライアント提供資料・既存成果物
- Output/: 全成果物
- Output/議事録/: 定例MTG・社内MTGの議事録
- Output/調査/: デスクリサーチ・分析結果
- Work/: 仕掛中のファイル・メモ等
- Meeting/: 会議資料・討議ドキュメント

## プロジェクト知識の現状
支援状況・課題・施策進捗等は [Output/プロジェクトサマリ.md](Output/プロジェクトサマリ.md) を参照。
```

## 状態管理の方法

活動カタログの各行はステータスを持たない（順序・進捗の概念がないため）。状態は以下の2層で管理する。

| 対象 | 管理方法 |
|------|---------|
| 単発で完結する活動 | 完了時に履歴ログへ1行追記するのみ |
| セッションをまたぐ活動 | 「進行中の活動」表でステータス管理し、完了時に表から削除して履歴ログに記録 |

「進行中の活動」のステータスは共通語彙を使う。

| ステータス | 説明 |
|------------|------|
| `in_progress` | 進行中 |
| `review_pending` | AIタスク完了、ユーザーレビュー待ち |
| `waiting_human` | 人間タスク待ち（クライアント返答待ち・外部パートナー待ち等） |

`not_started` の活動は表に載せない（カタログに存在すること自体が「起動可能」を意味する）。`completed` は表に残さず履歴ログで表現する。

## PM 呼び出し時の扱い

活動カタログ型プロジェクトでは、自動ルーティングの「現在のステップ」に相当するものがない。PM は以下を行う。

1. 「進行中の活動」表と履歴ログの直近数行を提示する
2. `review_pending` / `waiting_human` の活動があれば、通常どおりレビューゲート・完了報告の選択肢を提示する
3. なければ AskUserQuestion で「次に起動する活動」を活動カタログから選ばせる（月次サイクル活動の時期であればそれを最優先で提案する）

## 更新ルール

1. **活動の起動時**: セッションをまたぐ見込みなら「進行中の活動」表に追加（`in_progress`）
2. **AIタスク完了時**: ステータスを `review_pending` に更新し、レビューゲートへ
3. **レビュー承認時**: 表から削除し、履歴ログに成果物パスとともに追記
4. **知識状態の変化時**: SKILL.md の自動同期ルールに従い `Output/プロジェクトサマリ.md` を更新
5. **カタログの改廃**: 支援内容の追加・終了があれば活動カタログの行を追加・削除し、重要な意思決定に記録
