# 最終報告書作成スキル

デスクリサーチ・インタビュー等の調査結果を統合分析し、論点構造に沿った最終報告書を作成する。

## 入出力

- **入力（必須）**: 提案書（論点・小論点の構造定義）、報告書骨子（章立て・構成定義）、デスクリサーチレポート（1つ以上）
- **入力（任意）**: インタビュー議事録（0件以上）、インタビューまとめ、プロジェクトの CLAUDE.md
- **出力**: `Output/最終報告書.md`

## ワークフロー

**このスキルを使うときは必ず** `references/writing-guide.md` を読み、各層の書き方に従うこと。

### Step 1: 入力ファイルの確認

1. 提案書から大論点・小論点の構造を抽出する
2. デスクリサーチレポート、インタビュー議事録の一覧を確認する
3. CLAUDE.md があれば文書作成ガイドライン・スコープを確認する

### Step 2: 文書構成の確定

1. **デフォルト**: 提案書の大論点・小論点をそのまま報告書の骨格として採用する
2. **代替構成の検討**: 以下の条件をいずれか満たす場合、論点順以外の構成を採用してよい
   - 報告会の時間が限定的（1時間以下）で、論点順では読み手の理解負荷が高い
   - 論点間の論理接続が「現状→変化→選択肢→推奨」等の意思決定フローと自然に対応する
   - 代替構成の例: 競争ポジション→市場環境の変化→事業拡大の選択肢→推奨と実行要件
3. **代替構成を採用する場合の制約**:
   - 提案書の全小論点がいずれかの章でカバーされていることを検証する（→品質チェックG）
   - 小論点の見出しは問い形式を維持する（章の配列が変わっても小論点の粒度は変えない）
4. 調査過程で新たに判明した論点がある場合は、既存構造に統合する（勝手に論点体系を再構成しない）

### Step 3: ストーリーライン設計

個別の小論点を書き始める前に、文書全体のストーリーラインを設計する。

1. **全体の「問い→答え」を一文で定義する**: この調査が最終的に読み手に伝えるべきメッセージは何か
2. **大論点間の論理接続を設計する**: 大論点①→②→③が「なぜこの順番で並ぶのか」の理由を明確にする
  - 典型パターン: 外部環境→内部環境→戦略方向性 / 現状→課題→打ち手 / 市場→競合→自社
  - 各大論点の結論が次の大論点の前提になる「論理の連鎖」を意識する
  - **各接続部のブリッジ文を1文で設計し、ストーリーライン設計時に固定する**。ブリッジ文は「前章の結論が次章の問いをどう規定するか」を示す。例: 「第2章で差別化軸は運用品質・制度精通・ファイナンスと特定されたが、これらを自社単独で揃えるかエコシステムで補完するかが第3章の問いとなる」
3. **各大論点内の小論点の配列を検証する**: 小論点同士が「So What?」で上位メッセージに収斂するか確認する
4. **ストーリーラインを1段落で記述し、書き始める前に固定する**（出力ファイルには含めない）
5. **フレームワーク規律**: 分類フレームワーク（類型分け、マトリクス等）を用いる場合、以下を守る
   - 1つの大論点（章）で読み手が処理する分類フレームワークは原則1つに限定する。補助的な分類は主フレームワークの根拠として従属的に配置する
   - フレームワークの各カテゴリが、読み手の意思決定において異なる戦略的含意をもたらすか検証する（戦略弁別テスト）
   - 分類の全体集合（何をカテゴライズしているか、含むもの・含まないもの）を冒頭で明示する

### Step 4: 小論点ごとの分析（層3の作成）

各小論点を**メッセージ＋ボディ**の構造で記述する:

1. **メッセージ**（1-2文）: その小論点に対する端的な答えを冒頭に置く
2. **ボディ**: メッセージを支える論証を構造化して展開する
   - **サブヘディング**（主張文）で論理グループを構造化する。メタラベル（「ポイント」「整理」「概要」等）ではなく、そのグループが主張する内容を一文で表現する
   - 各グループ内でファクト・データ・表を階層化箇条書きで展開する
   - **含意**（So What?）で大論点サマリへの接続を示す

**論証の構造化**: 1セクションが2つ以上の独立した論証チェーン（それぞれ固有のファクト群→結論を持つ）を含む場合は分割を検討する。

**主張強度の適正化**: 根拠の強さに見合った主張表現を選ぶ。「希少」「唯一」「できない」等の強い修飾語を使用する際は、裏付けデータとの照合を行う。

情報源の参照（A01、B01等）は根拠の信頼度を示すために記載する。

### Step 5: 大論点サマリ・全体結論の作成（層2・層1の作成）

**ボトムアップで作成する**（小論点 → 大論点サマリ → 全体結論の順）:

1. 各大論点について、メッセージ（結論1文）＋ボディ（配下の小論点を統合した論証）で書く（層2）
2. 全体結論として、構造化箇条書きで書く（層1）
3. 横断的示唆を含む（層1の箇条書きの末尾、または独立した段落）

**メッセージ階層の役割分離**: 各層のメッセージは異なる役割を担い、同一内容の繰り返しを防ぐ。

- **層1（全体結論）**: 調査全体の最上位結論。ファクトと示唆を統合した完全な主張文で記述する。箇条書きの第1階層はトピック見出し（「競争環境」等）ではなく、それ自体が読み手に新しい知見を伝える文にする
- **層2（大論点サマリ）**: 配下の小論点を統合した上位主張。層1のメッセージより詳しいが、個別の小論点メッセージの繰り返しではない
- **層3（小論点）**: 個別の論証から導かれる結論＋根拠の完全な記述。ここが各結論の「正」の記述箇所となる

**対立見解の扱い**: インタビュイー間で見解が分かれた場合は、対立見解を構造的に提示し（対比表、論点別整理等）、統合的な含意を添える。

**検証**: 完成後に、同一の結論文が文書内に3回以上出現していないか確認する。3回以上出現する場合は、層3の論証箇所を「正」とし、層1・層2は参照（「第1章で確認したとおり」等）に書き換える。

### Step 6: 報告書としての仕上げ

報告書骨子の構成を参照し、以下を整備する:

1. **エグゼクティブサマリーの調整**: 層1（全体結論）を経営者向けの粒度に調整する。報告会で読み上げられることを想定し、専門用語の補足や文脈の追加を行う
2. **本編と付録の判定**: 「この表/データを除外したとき、読み手が結論の根拠を見失うか」で判定する。見失うなら本編に残し、見失わないなら付録に移す
3. **情報保全の確認**: ボディ内の定量データ・固有名詞・比較構造・出典が保持されているか確認する。削除してよいのは散文的な接続表現のみ
4. **付録の整備**: 用語集、参考文献一覧、データソース詳細、インタビュー対象者一覧（匿名化済み）等を必要に応じて追加する
5. **報告書骨子との整合確認**: 報告書骨子で定義された章立て・想定アウトプット（図表・マトリクス等）が網羅されているかを確認する

### Step 7: 品質チェック

本スキル末尾の品質チェックリストに基づき、全項目を確認する。違反があれば修正する。

## 文書構成: 3層のピラミッド

最終報告書は以下の3層で構成する。上の層ほど抽象度が高く、下の層が根拠・詳細を担う。

```
┌──────────────────────────────────────────────┐
│  層1: 全体結論（エグゼクティブサマリ）          │  ← 1回のみ
│    ① 大論点①の結論（2-3文）                    │
│    ② 大論点②の結論（2-3文）                    │
│    ③ ...大論点の数だけ繰り返し                  │
│    横断的示唆（1段落）                          │
├──────────────────────────────────────────────┤
│  層2: 大論点①サマリ                            │  ← 大論点の数だけ繰り返す
│  ┌────────────────────────────────────────┐   │
│  │  層3: 小論点①-1 の分析                  │   │  ← 小論点の数だけ繰り返す
│  ├────────────────────────────────────────┤   │
│  │  層3: 小論点①-2 の分析                  │   │
│  └────────────────────────────────────────┘   │
├──────────────────────────────────────────────┤
│  層2: 大論点②サマリ                            │
│  ┌────────────────────────────────────────┐   │
│  │  層3: 小論点②-1 の分析                  │   │
│  ├────────────────────────────────────────┤   │
│  │  層3: 小論点②-2 の分析                  │   │
│  └────────────────────────────────────────┘   │
├──────────────────────────────────────────────┤
│  層2: 大論点③サマリ                            │
│  │  ...同様に繰り返し                          │
└──────────────────────────────────────────────┘
```

読み手はどの層から読んでも必要な粒度の情報を得られる設計とする。

## 共通ライティング原則（全セクションに適用）

- 読み手はクライアント自身。「貴社」で書く。作業プロセスへの言及は削除
- 曖昧な定性表現ではなく具体的な数字・固有名詞・時間軸で記述
- すべての記述が「So What?」に答えている。整理・把握で終わらない
- ソースにない数字・事実を捏造しない。不明は「不明」と明記
- 略語は初出展開。分類・専門用語はその場で定義。結論文に主語を明記

## 9つの原則

以下は最終報告書固有の原則:

### 原則1: 結論を先に述べ、根拠で支える

各レベル（全体結論・大論点サマリ・小論点）において、結論を冒頭に置き、それを支えるファクト・データ・表を続ける。積み上げ型（根拠→結論）ではなく、トップダウン型（結論→根拠）で書く。

### 原則2: すべての記述が「So What?」に答える

文書内のすべてのパラグラフ・すべてのデータは「それで、何が言えるのか？（So What?）」に答えなければならない。

- **ファクトの羅列で終わらない**: データや事実を提示した後、必ず「→ したがって〜」「→ これは〜を意味する」の一文を添える
- **So What? の連鎖で上位に接続する**: 小論点の含意 → 大論点サマリの根拠、大論点の結論 → 全体結論の根拠、という連鎖が途切れないようにする
- **逆方向の検証（Why So?）**: 結論に対して「なぜそう言えるのか？」と問い、根拠が不足していないか確認する
- **検証の目安**: 各小論点の「含意」を読むだけで、大論点サマリのメッセージが自然に導出できるか。できなければ含意の記述が弱い

### 原則3: 仮説検証の作業プロセスを残さない

アウトプットには結論とその根拠のみを記載する。以下の要素は文書に含めない:

- 「**仮説**: ...」「**検証結果**: ...」等のラベル
- 「修正が必要」「部分修正」「支持」「部分的に否定」等の検証ステータス
- 「提案書で設計した仮説では〜」等の作業プロセスへの言及
- 「キックオフMTGで〜氏が指摘」等の社内イベントへの言及

ただし、結論の確度に関する留保（「本分析は○○を前提としており、適用にあたっては前提条件の読み替えが必要」等）は、結論の一部として許容する。

### 原則4: ソース別列挙ではなく論点別統合

デスクリサーチの知見とインタビューの知見をソース単位で列挙しない。各小論点の中で、出所の異なる情報を結論と根拠の関係で一体的に記述する。

- ✕ 「デスクリサーチからの知見: ...」「インタビューからの知見: ...」と分けて列挙
- ○ 結論を述べ、その根拠としてデスクリサーチ・インタビュー双方のファクトを織り交ぜて記述

インタビュー対象者間で見解が分かれた場合は、対立見解として整理し、それぞれの論拠を明示する。

### 原則5: 読者の前提知識に依存しない

文書のどの箇所でも、読者がその箇所に到達した時点で持っている知識だけで理解できるように書く。「読者が持っていない前提知識」には、用語・文脈だけでなく、書き手の頭の中にある推論の構造（根拠同士の役割分担やストーリー）も含まれる。

- **用語・枠組みの事前導入**: 分類枠組みや専門用語を使う場合、その箇所の中で最低限の定義を与える。上位層（層1・層2）は下位層をまだ読んでいない読者が対象であり、層3でも前の小論点の内容を前提としない
- **結論の主語明示**: 結論文には「誰にとっての」示唆かを必ず明記する。市場一般のトレンドなのかクライアント固有の示唆なのかを、文面だけで判別できるようにする
- **根拠の論理的役割の可視化**: 複数の根拠を列挙する場合、各根拠が結論をどの角度から支えているか（必要性/実現可能性/固有優位性、現状の制約/打開条件/推奨根拠 等）を接続語またはラベルで明示する。フラットな箇条書きで論理構造を潰さない
- **略語の初出展開**: 略語は初出時に `正式名称（略称）` の形式で展開し、2回目以降は略称のみ使用する。一般ビジネス用語（IT、AI、API、SaaS、M&A等）は展開を免除するが、迷う場合は展開する
- **参照の初出説明**: 図表名・分析フレーム名・内部参照番号は、初出時に括弧書きまたは1行で内容を補足する

### 原則6: 意外な結果を積極的に取り上げる

通説・直感に反するファクト（例: 高年収層ほど非年収要因を重視する）は報告書の価値を高める最大の材料である。意外な結果が見つかった場合は積極的にSo What / Why Soを展開し、読み手に新しい気づきを提供する。「想定どおりの結果」よりも「想定に反する結果」の方が示唆に富む。

### 原則7: 事実と仮説を文体で区別する

スコープ外の論点に関する知見は、断片的なデータに基づく仮説であることを文体で自然に区別する。

- 確定事実: 「〜である」「〜が確認された」
- 解釈・推論: 「〜と考えられる」「〜の傾向が見られる」
- 仮説: 「〜の可能性がある」「〜し得る」

「仮説:」「検証結果:」等の作業ラベルは使わない（原則3と連動）。読み手は文体の違いで確度を自然に受け取れる。

## 品質チェックリスト

文書完成後、以下を確認する。

### A. 構造・ピラミッド

- 全体結論の番号付けが本文の大論点と1対1で対応しているか
- 全体結論に横断的示唆（論点間の統合メッセージ）が含まれているか
- 全体結論の箇条書き第1階層が完全な主張文になっているか（「競争環境」等のトピックラベルではないか）
- 各大論点がメッセージ（結論1文）＋ボディの構造で書かれているか
- 各小論点がメッセージ→ボディの順で書かれているか
- 1セクションに独立した論証チェーンが2つ以上混在していないか

### B. ストーリーライン・論理接続

- 大論点の配列に論理的な必然性があるか（なぜこの順番なのか説明できるか）
- 各セクションのメッセージを順に読んだとき、論理の連鎖が途切れないか
- 各小論点の含意を読み上げるだけで、大論点サマリのメッセージが自然に導出できるか（So What? の連鎖）
- 全体を通して一つの明確なストーリー（問い→分析→答え）が読み取れるか
- 使用しているフレームワークの各カテゴリが異なる戦略的含意をもたらすか（戦略弁別テスト）

### C. 可読性・自己完結性

- 各記述で使われている専門用語・枠組みが、その記述に到達する前（または同一セクション内）で定義されているか
- 結論文に主語（クライアント名・「貴社」等）が明示されているか。市場トレンドとクライアント固有の示唆が文面上区別できるか
- 根拠が3つ以上並ぶ場合、各根拠の論理的役割が接続語・ラベル・階層構造のいずれかで区別できるか
- すべての略称が初出時に展開されているか
- 図表名・内部参照番号が初出時に説明されているか

### D. メッセージの切れ味

- すべてのパラグラフが「So What?」に答えているか（ファクトの羅列で終わっていないか）
- 結論に対して「Why So?」と問い返したとき、根拠が十分に示されているか
- 曖昧な定性表現（「多い」「高い」「拡大傾向」）が具体的な数字・事実に置き換えられているか
- 結論が平凡な一般論ではなく、この調査固有の発見・示唆を含んでいるか
- 否定形の結論（「AはBではない」「Aにとどまらない」）で止まっていないか。「つまりCである」まで書き切っているか
- 「最多級」「主要な」等の曖昧な相対表現で逃げていないか。データを持っているなら正確な順位・数値で記述しているか
- 代替解釈（別の読み方）が存在する場合に一面的な結論を避けているか。代替解釈を排除できる根拠があればそれを明記し、排除できなければ両論を提示しているか
- 通説・直感に反する結果（意外なファクト）を見逃さず、So What / Why Soを展開しているか

### E. 形式・表現

- 「仮説」「検証結果」等の作業プロセスラベルが残っていないか
- 「修正が必要」「支持」等の検証ステータスが残っていないか
- 「クライアント」「先方」「お客様」等の社内目線表現が残っていないか
- 冒頭説明文が結論中心の記述になっているか
- ソース別の知見列挙（「デスクリサーチからの知見」「インタビューからの知見」）ではなく、論点ごとに統合された記述になっているか
- 冗長な修飾・回りくどい表現がなく、一文一義で簡潔に書かれているか
- 章タイトルにダッシュ（—）区切りの副題がないか。ある場合は統合して1フレーズにする
- 大仰・ドラマチックな動詞（「突きつける」「切り拓く」「紐解く」「浮き彫りにする」等）がないか
- AI典型表現（「構造的な帰結」「足場」「閾値の構造」「空白がもたらす」等）がないか
- クライアントに馴染みのない英語表記（「Key Takeaway」「Implications」等）がないか。日本語に置き換えているか
- 統計用語（「二峰性」等）を平易な日本語で代替可能な場合に使っていないか
- カジュアルすぎる口語（「ピッタリ」等）や投げやりな表現（「〜は別問題」等）がないか
- 冒頭に「以下で詳しく見ていきます」「順を追って整理します」等の案内的表現がないか
- 同じ構造の文頭が3回以上連続していないか（「まず〜」「次に〜」「最後に〜」等の機械的パターン）
- 箇条書きの各項目が同一の文型（「〜が重要である」「〜が必要である」の反復）で終わっていないか

### F. ファクト整合性

- 強修飾語（「希少」「唯一」「初」「最大」「不可欠」「できない」等）の使用箇所で、裏付けデータとの照合が行われているか
- マトリクス・分類表の内容と本文の記述が矛盾していないか（例: マトリクス上で同一セルに3社が存在するのに「希少」と記述していないか）
- 定量推計の前提が、文書内の他の主張と矛盾していないか（例: 「SWコモディティ化が進行」と述べつつ、TAM推計で高価格水準を前提としていないか）
- 「参入障壁は低い」と述べつつ、別の箇所で「模倣困難な蓄積」を差別化要因として挙げていないか（障壁の対象が異なる場合は、何の障壁かを明記する）
- 時間軸の矛盾: 「短期的には困難」と述べた事項を、時間軸アクション表の短期欄に配置していないか
- 選択肢を任意に括って「最大」「過半」等を主張していないか。括る場合はその根拠（連続区間、同質グループ等）を示しているか
- 同一概念の数え方（N個の○○）が文書全体で統一されているか。サマリ・本文・図で数字が不一致になっていないか
- フレームワークのラベル（「層」「段階」「類型」等）が概念の実態と一致しているか。上下構造でないものを「層」と呼んでいないか、順序のないものを「段階」と呼んでいないか

### H. テキストと図表の整合

- ボディで主張するデータポイントが同一セクションの図表に存在するか
- 図表にないデータポイントをボディに書いていないか（図で触れていない要素をボディに書くとメッセージが濁る）
- テキストで「N社」「N件」と言及した場合、テーブルにN行あるか
- パーセンテージの合計が100%になっているか（端数処理による誤差は最大剰余法で調整）
- 図表のタイプ名称（ヒートマップ、散布図等）と実際の可視化が一致しているか

### G. スコープ整合性

- 提案書の全小論点に対し、報告書のどのセクションがカバーしているかをマッピングしたか
- カバーされていない小論点がある場合、以下のいずれかの処理を行っているか:
  - (a) 該当セクション内で「本調査では情報を取得できていない」と明記する
  - (b) 調査結果が小論点への回答そのものである場合、その旨を明示する（例: 「SLA/ペナルティは業界標準として設計されておらず、免責ベースが定着している」が①-2への回答である旨）
- 提案書の論点に対応しない独自分析がある場合、「調査結果から自然に導出された知見」か「独立した追加分析」かを判別し、後者はCLAUDE.mdのスコープ制約に照らして含めるか判断しているか
