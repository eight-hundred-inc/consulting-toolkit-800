# 各層の書き方ガイド

## 層1: 全体結論（エグゼクティブサマリ）

**役割**: 文書全体のメッセージを凝縮する。読み手がここだけ読んでも調査の主要な結論が把握できる。

### 構造化箇条書きで構成する

全体結論は構造化箇条書きで記述する。箇条書きの各項目が調査の主要な結論を伝え、全体として一つのストーリーを形成する。

**箇条書き設計原則**:

- **第1階層はファクトと示唆を統合した完全な主張文にする**。「競争環境」「規制変更」等のトピックラベルを第1階層に置かない。読み手が第1階層だけ拾い読みしても調査の結論が把握できる状態にする

```
✕ - **競争環境**
  - XX市場では成長企業がサービス兼営モデルに集中している...

○ - XX市場では成長企業がいずれもサービス収益を兼営しており、PF専業モデルの
    成功は限定的である。貴社が属する「PFあり×サービスなし」の象限からの脱却が
    事業拡大の前提条件となる
```

- 結論で使う分類・枠組みの名称（例: 「3類型」「2軸」等）は、本文を先読みしなくても意味が取れるよう1行で補足する
- 結論文には必ず主語を入れる（「貴社にとって〜」「○○市場では〜」）。市場全体のトレンドなのかクライアント固有の示唆なのかを文面だけで判別できるようにする
- 横断的示唆（論点を横断して統合した結論）を箇条書きの末尾、または独立した段落として含める。大論点の中に「統合的な結論」「戦略オプション」等の総括セクションがある場合は、その存在を明示して参照する形でもよい

### サイズ制約

- 層1の目標は文書全体の5-8%以内とする。500行の文書なら25-40行が目安
- 大論点が4つ以上ある場合、各結論を1-2文に圧縮し、層1が肥大化しないよう制御する
- 層1で根拠を展開しない（SKILL.md Step 5 の反復排除ルールと連動）。根拠は「→第X章参照」で委ねる
- 層1が目標を超過した場合、根拠の展開が混入していないか、横断的示唆が冗長でないかを確認する

## 層2: 大論点サマリ

**役割**: 各大論点の見出し直後に置き、配下の小論点すべてを統合した上位の結論を示す。

### 書き方: メッセージ＋ボディ

大論点サマリは**メッセージ＋ボディ**の構造で書く。

- **メッセージ**: 大論点全体を通じて得られた結論を1文で述べる。小論点の内容を列挙・要約するのではなく、小論点を統合して得られる上位主張を書く
- **ボディ**: メッセージを支える統合的な論証を構造化箇条書きで展開する。読み手がサマリだけ読めば大論点の結論とその骨格がわかり、小論点はその根拠・詳細として位置づけられる構造にする
- **枠組みの導入**: 結論が類型・分類・フレームワークに依存する場合、結論文の前に1-2行で枠組みを提示する。読者はまだ下位の小論点を読んでいないため、結論を受け取るための最低限のボキャブラリーをここで渡す
- **用語の初出定義**: サマリ内で初めて登場する専門用語・略語は、`正式名称（略称）` の形式で展開する
- **主語の明示**: 結論文には必ず主語を入れる。「○○型への移行が収益構造上の自然なパスとなる」ではなく「貴社にとって○○型への移行が〜」と書く

### ダミー例

```markdown
## 大論点① 競合他社の事業モデルと収益構造

**問いは「保全サービスに関与するか否か」ではなく「どこまで、どのような形で関与するか」である**

対象市場のプレーヤーを「PF/SaaS提供の有無」と「自社保全サービスの
有無」の2軸で整理すると、成長企業はいずれもPF＋保全サービス収益を
兼営している。

- 貴社（AXCEL）が位置する「PFあり×保全なし」の象限では、SaaS単体の
  収益はサービス層の3分の1以下にとどまり、PF専業の成功は限定的である
- HW自前は現時点で数少ない差別化要素だが、汎用センサーの精度向上が
  進めば優位は薄まる方向にある
- → 貴社の戦略課題は「マトリクス上で保全サービス兼営の象限へ移動
  するか否か」ではなく、「どのような形で、どこまでサービスレイヤーに
  関与するか」にある

### 小論点①-1: 主要な収益源はどこにあるか

（ここから層3の記述が始まる）
```

## 層3: 小論点の分析

**役割**: 個別論点ごとの結論と、それを支えるファクト・データを一体的に記述する。

### 構成パターン: メッセージ＋ボディ

```markdown
### 小論点X-Y: 問いの形式で書く

**メッセージ**: 結論を1-2文で端的に述べる。

**サブヘディング（主張文）**

- 根拠となるファクトを階層化箇条書きで展開する
    - 具体的なデータや事例（情報源ID）
    - 補足的な定量情報・出典

| データ表がある場合 | ここに配置する |
| --- | --- |
| 項目A | 値A |
| 項目B | 値B |

→ ファクトから導いた推論・含意（So What?）
```

### 根拠の構造化

#### サブヘディングは主張文にする

根拠を構造化する際のサブヘディング（太字見出し）は、そのグループが主張する内容を一文で表現する。メタラベルは禁止する。読み手がサブヘディングだけ拾い読みしても論理の流れが追える状態にする。

```
✕ **3つの読み取り**
✕ **ポイント**
✕ **市場の特徴**

○ **成長企業はPF＋サービス兼営モデルに集中し、PF専業は停滞している**
○ **センサーデータの精度問題がHW保有者の優位を生んでいる**
○ **規制変更は単一市場での収益モデルを困難にする**
```

#### 階層化箇条書きの構造

根拠が3つ以上のファクトを含む場合、フラットな箇条書きではなく、論理的グループごとに階層化する。

- **第1階層**: 主張または論理グループの見出し（完全文）
- **第2階層**: その主張を支えるファクト・データ
- **第3階層**: 補足的な定量情報・出典

#### サブヘディングとメッセージの非重複

メッセージ（冒頭の結論文）とボディ内のサブヘディングは異なる粒度を担う。メッセージは「このセクション全体で言いたいこと」、サブヘディングは「そこに至る論証ステップ」であり、同じ文言を繰り返さない。

#### セクション分割の判定

1つのセクションが2つ以上の独立した論証チェーン（それぞれ固有のファクト群→結論を持つ）を含む場合は分割する。判定方法: セクション内のサブヘディングを並べたとき、1つのメッセージに収斂しない場合は分割が必要。

#### 典型的な論理構造パターン

各根拠が結論をどの角度から支えているかを、接続語またはラベルで明示する。

- なぜ必要か（現状の制約・限界）→ 実現可能か（参入障壁・移行条件）→ なぜこのクライアントか（固有の優位性・差別化要因）
- 市場側の根拠（需要・規制・トレンド）→ 技術側の根拠（成熟度・差別化）→ 経済性の根拠（コスト・収益構造）
- 現状の制約 → 打開の条件 → 推奨アクション

```markdown
### 小論点①-4: 貴社の参入優位性はどこにあるか

**メッセージ**: 貴社はSW・HW（ハードウェア）双方を自社で保有する点で、
ハイブリッド型への移行において参入優位性を持つ。

**ライセンス単体では収益に構造的な限界がある**

- 市場の課金構造はベネフィットシェア型（運用収益の10-15%を報酬として
  受け取るモデル）が主流（①-1）
- PF利用料に対し、保全サービス料＋導入費が大幅に上回る構造

**契約面の参入障壁は低い一方、データ蓄積が差別化を左右する**

- 契約設計は免責ベースが業界標準であり参入障壁は高くない（①-2, ①-3）
- 制御精度のデータ蓄積が実質的な差別化要因となる

**SW領域のコモディティ化がHW保有者の優位を浮き立たせている**

- 主要14社の技術領域を比較した結果、SW領域のコモディティ化が確認された
- HWを自社製造する貴社の特性は、SW＋HW統合制御という参入障壁の高い
  領域で優位に立てる（①-4調査）

→ 貴社のHW接点は移行期における希少な資産だが、汎用センサーの精度向上が
進めば優位は時限的である。この猶予期間内にサービスレイヤーへの布石を
打てるかが分水嶺となる
```

### 小論点間の用語依存と略称展開

前の小論点で導入した概念・略語が次の小論点で登場する場合も、初出扱いで簡潔に補足する。読者が小論点を飛ばし読みする可能性があるためである。補足は括弧書き1つ程度で十分であり、冗長な再説明は不要。

**略称展開の書式**:
- 初出時: `正式名称（略称）` — 例: `投資対効果（ROI）`
- 2回目以降: 略称のみ — 例: `ROI`
- 文書全体を通じて一貫する。セクションをまたいでも再展開は不要（文書冒頭で1回展開すれば十分）

**展開免除の判定基準**:
- 読み手が当該業界の経営層であることを前提とし、一般ビジネス用語（IT、AI、API、SaaS、M&A等）は免除する
- 文書の主題として冒頭で定義済みの用語も免除（ただし冒頭での定義は必要）
- 迷う場合は展開する。展開による情報過多のリスクより、未展開による理解阻害のリスクの方が大きい

### 「含意」の書き方

含意は「So What? への答え」であり、小論点の分析を大論点の結論に接続するための最も重要なパラグラフである。

**良い含意の条件**:
- この調査・分析固有の発見から導かれた、読み手にとって新しい示唆を含む
- 読み手の意思決定や行動に直結する具体性がある（「何をすべきか」「何を避けるべきか」が見える）
- 上位の大論点サマリのメッセージを構成する「部品」として機能する

**避けるべき含意**:
- ✕ 「今後の動向を注視する必要がある」（一般論で行動を示唆しない）
- ✕ 「XXは重要な要素である」（重要だと言うだけで何も伝えていない）
- ✕ 「慎重な検討が求められる」（検討の方向性が不明）
- ○ 「貴社が運用レイヤーに参入するなら、A社の価格帯（XX万円/拠点）を下回る設計が必須となる。ただしXX以下では収益性を確保できないため、対象は年間XX拠点以上の大口顧客に限定すべきである」

### ダミー例

```markdown
### 小論点①-1: 主要な収益源はどこにあるか

**メッセージ**: SaaS/ライセンス収入は保全サービス料の3分の1以下で
あり、収益の主体はサービスレイヤーにある。

**課金モデルは成果連動型が主流で、保全サービスに紐づく構造が定着
している**

- 成果連動型（ダウンタイム削減額の15-20%）が主流（A01）
- A社は固定＋従量＋成果報酬のハイブリッド課金であり、市場では
  例外的な存在（C01）

**A社の料金体系が示す収益格差**

| 項目 | 金額 |
| --- | --- |
| PF利用料（基本） | 120万円/年 |
| PF利用料（従量） | 3万円/接続ポイント |
| 予知保全サービス（基本） | 360万円/年 |
| 予知保全サービス（従量） | 8万円/設備 |
| 導入・チューニング費 | 300万円/拠点 |

- PF利用料に対し、保全サービス＋導入費が大幅に上回る

→ SaaS単体にとどまれば、サービスレイヤーの収益を取りこぼす
ことになる。貴社がPF提供のみで収益を確保するのは構造的に困難であり、
何らかの形でサービスレイヤーへの関与が必要となる
```

### 表の活用指針

- 費用体系・比較・制度変更一覧等は表で視認性を確保する
- 単一ソースのファクト表（上記例のような費用体系表）に加え、複数ソースの横断比較表（例: 競合3社を同一観点で並べて比較する表）も有効
- 表の前後に、表が示すポイントを文章で補足する（表だけで結論を語らない）

## 総括大論点の扱い

大論点構造の中に「戦略オプション」「統合的な結論」等の総括的な大論点がある場合、その大論点は他の大論点の結論を横断的に統合する役割を持つ。この場合:

- 全体結論の横断的示唆は、この総括大論点の結論を要約・参照する形で書く
- 総括大論点のサマリには、結論文を冒頭に置く（「XX方向を優先し、YY方向は既存市場の深耕に集中する」等）

### 戦略オプションを含む場合の必須記述項目

戦略オプション（A/B/C/D等の選択肢）を提示する場合、各オプションに以下の3要素を必ず含める:

1. **(a) 具体像とトレードオフ**: 何をするのか、何を得て何を失うのか
2. **(b) 必要な体制・リソース・パートナーシップ**: 実行に必要な人材・組織体制・外部パートナーを明示的に切り出す。トレードオフの記述に埋没させない
3. **(c) 一次情報に基づく評価**: インタビュー等で得られた実務者の評価・見解。見解が分かれる場合は対立見解として両論を提示する

(b)はトレードオフ記述の中に「運用人材の確保が必要」等として埋もれやすい。独立した項目として切り出し、読み手が各オプションの実行条件を一覧で比較できるようにする。

## 情報源の参照ルール

### 参照形式

- **単独参照**: `（A01）`、`（B01）` -- 特定の情報源を根拠として示す
- **複合参照**: `（A01＋デスクトップ調査）` -- 複数の異なるソースが同一の結論を支えている場合に使用し、根拠の厚みを示す

### 情報源IDの体系

プロジェクトのインタビュー議事録・デスクリサーチレポートで使用されているID体系をそのまま踏襲する。

### 仮説ラベルの判断基準

排除すべきもの（作業プロセスのラベル）:
- 「仮説: ○○」「検証結果: 支持/棄却」
- 「修正が必要」「部分的に支持」
- 「提案書での仮説に対し〜」

許容されるもの（結論の確度に関する留保）:
- 「本分析は○○を前提としており、適用にあたっては前提条件の読み替えが必要」
- 「○○については調査対象が限定的であり、追加検証を推奨する」
- 「○社の事例から抽象化した枠組みであり、一般化には留意が必要」

判断基準: **作業の経過を報告しているか、それとも結論の信頼性について読み手に有用な情報を提供しているか**。前者は排除、後者は許容する。

## フレームワーク設計ガイド

分類フレームワーク（類型分け、マトリクス等）を使う際の設計指針。SKILL.md Step 3「フレームワーク規律」の詳細版。

### 戦略弁別テスト

フレームワークの各カテゴリが、読み手の意思決定において異なる戦略的含意をもたらすかを以下の手順で検証する。

1. 各カテゴリに対して「このカテゴリに属する場合、読み手は何をすべきか」を1文で書く
2. 複数のカテゴリが同一の推奨アクションに帰結する場合、そのフレームワークは粒度が細かすぎる。統合するか、より弁別力のある軸に再設計する
3. 同一カテゴリ内に成功企業と停滞企業が混在する場合、そのフレームワークの説明力は低い

### 全体集合の明示

分類を用いる際は、何をカテゴライズしているのか（全体集合の定義）を冒頭で明示する。全体集合の境界（含むもの・含まないもの）を1文で定義する。

### 補助的な分類の扱い

補助的な分類（技術マップ等のファクトデータ）は、主フレームワークの根拠として従属的に配置し、独立したフレームワークとして提示しない。1つの大論点（章）で読み手が処理する分類フレームワークは原則1つに限定する。

## 表と箇条書きの使い分け

### 表を使う場合

複数の対象を同一の観点で横並び比較するとき。競合比較、料金体系、オプション比較、時間軸別アクション等。

### 箇条書きを使う場合

1つの主張を複数の根拠で支えるとき、または論理の流れ（順序・因果）を示すとき。

### 判定基準

迷う場合: 「列見出しと行見出しの両方を定義できるか」。定義できるなら表、できないなら箇条書き。

## 主張強度の制御

主張を記述する際、根拠の強さに見合った表現を選択するための参考指針。

| 強度 | 表現例 | 使用条件 |
|---|---|---|
| 強 | 「〜である」「〜できない」「唯一の〜」 | 複数の独立した根拠で直接確認されている |
| 中 | 「〜と考えられる」「構造的な限界がある」「〜の傾向が見られる」 | 1つの根拠で確認、または推論を含む |
| 弱 | 「〜の可能性がある」「〜し得る」 | 間接的な根拠からの推論 |

「希少」「唯一」「最大」「不可欠」「できない」等の強修飾語を使用する際は、裏付けデータ（マトリクス・表中の該当数等）との照合を行う。

## 文章レベルの品質基準

一流コンサルティングファームの報告書に求められる文章品質を満たすためのガイドライン。

### 一文一義

- 一つの文に複数のメッセージを詰め込まない
- 「〜であり、〜だが、〜ため、〜である。」のような長い複文を避け、短い文に分割する
- 目安: 1文は60-80字以内。100字を超える文は分割を検討する

### 主語と述語の対応

- 主語と述語の距離を近づける。修飾語句が間に入りすぎると論旨がぼやける
- 「〜は、…（長い修飾）…、…（さらに修飾）…、○○である。」→ 修飾部分を別文にする

### 対比・比較で切れ味を出す

ファクトを並べるだけでなく、対比構造を活用してメッセージを鋭くする。

- 「A社はXX方式を採用している」→「A社はXX方式を採用する一方、B社はYY方式を選択しており、この違いは○○に起因する」
- 対比は「違い」だけでなく「なぜ違うのか」「その違いは読み手にとって何を意味するか」まで踏み込む

### 能動態を基本とする

- ✕ 「市場の拡大が見込まれている」（受動態・主体不明）
- ○ 「XX調査機関は市場が年率X%で拡大すると予測している」（主体明確）
- ✕ 「導入が推奨される」
- ○ 「貴社はXXの導入を優先すべきである」

### 抽象語の空転を避ける

以下のような表現は、具体的な内容に置き換えるか、削除する:

- 「最適化」→ 何をどう変えるのか具体的に書く
- 「シナジー」→ どの事業とどの事業が、何をどう共有・連携するのか
- 「DX推進」→ どの業務プロセスを、どの技術で、どう変えるのか
- 「戦略的に重要」→ なぜ重要なのか、何に対して重要なのかを書けば「戦略的に」は不要になる
