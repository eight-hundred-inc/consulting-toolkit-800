# 共通ライティング原則

クライアント向け文書（提案書、最終報告書等）に共通して適用する品質原則。各スキルの固有ルールはスキル側に定義する。

---

## 原則1: 読み手はクライアントである

文書の読み手はクライアント自身であることを常に意識する。

- 「クライアント」「先方」「お客様」→ 社名または「貴社」で記載する
- 自社の作業プロセス（提案書の仮説設計、キックオフでの議論等）への言及は削除する
- 冒頭説明文も「仮説の検証結果と結論を示す」等ではなく「分析結果と示唆を示す」とする

## 原則2: 具体性と定量化を徹底する

曖昧な定性表現を排し、可能な限り具体的な数字・固有名詞・時間軸で記述する。

- ✕ 「市場は拡大傾向にある」「多くの企業が導入している」「コストが高い」
- ○ 「市場規模は2023年の約XX億円から年率X%で成長し、2027年にはXX億円に達する見通し」
- ○ 「導入企業はA社・B社・C社を含む約XX社に上る」

定量データが入手できない場合は「推定XX〜XX」「業界関係者によればおおむねXX規模」等のレンジ表記や定性的な根拠の明示で代替し、数字なしの印象論は避ける。

## 原則3: So What? を常に問う

- 「把握する」「整理する」「深掘りする」だけで終わる文は禁止
- メッセージ行は結論・打ち出しを書く（「把握する/整理する」ではなく「統合して絞る/突破口を提示」）
- 読み手がすでに知っている情報の再整理にとどめない。新たな切り口・示唆を提供する

## 原則4: ファクトの正確性

- 入力情報に存在しない数字・固有名詞・事実を捏造しない
- ソースから引用した数字はそのまま使用し、加工した場合は加工方法を明記する
- 不明な点は「不明」「要確認」と明記し、推定で埋めない
- **すべてのファクト記述には出典URLを添える**。URL がないものは「ファクト」として扱わない（推測・仮説として明示する）
  - 例: 「市場規模 XX 億円（出典: https://…）」
  - 例: 「A 社は XX を導入（出典: 公式プレスリリース URL、要約: 〇〇）」
- 出典 URL は **存在確認をしたうえで** 記載する（404 / リダイレクト先が無関係 / 推測で生成した URL は不可）
- 主要ファクト出典は文書末尾に**台帳形式**で再掲する（脚注番号と URL の対応表）

## 原則5: 読者の前提知識に依存しない

- 略語は初出時に `正式名称（略称）` で展開する（一般ビジネス用語は免除）
- 分類枠組みや専門用語を使う場合、その箇所で最低限の定義を与える
- 結論文には「誰にとっての」示唆かを明記する

## 原則6: 本文は「読み手の意思決定に必要なもの」だけで構成する

クライアント向け文書の本文には、読み手の意思決定を進める論点・分析・示唆だけを書く。作成者側の事情・営業文脈・実施済の経緯・自己言及は、本文から排除する。

**判断基準**: 「この文を消して、読み手の意思決定に必要な情報が欠けるか？」を問う。欠けないなら削る。

### 排除すべきカテゴリ

**作成プロセスのメタ言及**: 書き手がどう構成したか、どんな前提で書いたかは本文に書かない。構成は読み手が読めば分かる。
- ✕ 「与件で示された4テーマ・10項目を、意思決定に直結する『問い』として再構成する」
- ✕ 「事実と仮説を明示的に切り分ける構成原則のもと」

**既往実績・経緯への自己言及**: 過去の作業や経緯は本文に持ち込まない。引用が必要なら根拠として一行添える。
- ✕ 「探索的調査（実施済）の結果を踏まえて」
- ✕ 「Phase 1で分かったことをもとに」（見出しとしての使用も含む）

**営業章・営業フレーズ**: 提案書の本文に売り込み章・自社売り込みフレーズを挿入しない。体制やスケジュールは事実として淡々と記載する。
- ✕ 「中間共有・中間報告」「予算内で品質を担保するための工夫」を独立章として立てる
- ✕ 「貴社が独自に持つ資産を活かして8週間で成果を出す」

**スコープ外の明示的羅列**: 「除外」と書いて長く列挙すると、読み手は「やらないこと」に意識が引っ張られる。スコープ内を書く。本当に紛らわしい1〜2点だけ短く除外明示する。
- ✕ 「スコープ外: 個別ベンダーの詳細PoC実装支援／貴社固有環境への適用設計／…（長い羅列）」

**口語・煽り・AI臭表現**: プロが書く文書として違和感のある表現を避ける。
- ✕ 「そのまま踏襲し」「主に効く論点」「白地図」「シニア」「意思決定に効く」
- ✕ 「立ち上がる」（→ 「合意できた」「定着しつつある」「形になりつつある」など状態語に置換）
- ✕ 「手応えがある」（主体が我々ではなくクライアント。「クライアントから一定の反応があった」など事実記述に）
- ✕ 「中堅ERPベンダーというラベルによる足切りを跳ね返すか？」（煽り）
- ✕ 略号の独断展開・独断省略（「NSM」「L2」など、本文中で初出する略号は与件・社内文書・公的資料に既出かを確認し、ない場合はフル展開で書く）
- ✕ 箇条書きの中に「見出しっぽい一行（体言止め＋鍵括弧）」を入れない（例: 「一次情報の中にも価値の段差がある」のような名詞句行）。箇条書きは「主語＋述語」の完結した一文にする
- ✕ 「クリスタライズして」「凝集して」「圧縮して」「短く」と指示されたのに、同義語の追加で行数を増やす（依頼に反する展開）。原文より短くすることが必須
- ✕ 装飾だけの比喩（「情報の海」「〜の旅」等）。意味を運ぶ中心概念の言い換えは可だが、装飾目的の比喩・大仰な比喩名詞/比喩動詞は機能語に置換する（比喩名詞・比喩動詞・物理メタファ動詞・戦闘メタファ等の詳細リストは `integrated-analysis-creator/references/writing-guide.md`「AI典型表現の回避」および各スキルの品質チェックリストを参照）
- ✕ 詩的・ナラティブ的なつなぎ／余韻（「ひとつの発見が次の問いにつながっていく」のような物語調の接続・余韻、「『できるはず』ではなく『もう動いている』」のような自己効力アピールの対比）。即物的に事実と示唆だけを書く
- ✕ 本文中のダッシュ（——／ーー）での挿入・言い換え・余韻（「──このあたりが数字で表れる」等）。読点・句点で区切るか文を分ける
- ✕ 「詰まり」等の口語接続、「効きどころ」「見え方」のような名詞化された曖昧語、「（型を）配る」のような物理メタファ動詞
- ○ 自然なビジネス日本語、与件のニュアンスに沿ったプレーンな表現

## 原則7: NG語彙・言い換え辞書

案件を跨いで再発した差し戻し語彙をここに集約する。**ユーザーから表現の差し戻しを受けたら、この辞書に追記する**（同じ指摘を別案件で繰り返させない）。原則6・検査基準A/E と重複する語（収斂／最適解／決定打／情報の海／効きどころ／見え方／詰まり／立ち上がる 等）はそちらが正本なので再掲しない。以下は未収録の追加分。起草時と提出前の両方で grep 突合する。

**AI臭・比喩系**（装飾比喩・造語。機能語に置換する）:

| NG | 方針・理由 |
|---|---|
| 試金石 | 大仰な比喩。「判断材料」「検証対象」等に置換 |
| 景色（〜の景色が変わる） | 比喩名詞。何がどう変わるかを即物的に書く |
| 溶かす（境界を溶かす等） | 物理メタファ動詞。「なくす」「統合する」に置換 |
| 効く／効きどころ | 曖昧な効能語。「〜に有効」「重点」と具体化 |
| 重心／収斂 | 比喩。「主眼」「集約」等の機能語に |
| ポエム調の言い回し | 詩的余韻は排除し、事実と示唆だけ書く |
| 「4本柱」等の造語ラベル | 安易な数値ラベリングを作らない。実体の名詞で列挙 |

**ビジネス硬直・意味の薄い抽象語**（中身を名指しする）:

| NG | 方針・理由 |
|---|---|
| 方向感 | 「方針」「見通し」と言い切る |
| 含意 | 「示唆」「意味するところ」に。多用しない |
| 現実解 | 「実行可能な案」等、具体で書く |
| 核となる／中核になる | 「主要な」「中心的な」。造語的な格上げを避ける |
| 出自／系統 | 「出所」「種類」等の平易語に |
| 射程（〜の射程） | 「範囲」「対象」に。比喩用法を避ける |
| 本番化 | 「本格運用」「実運用への移行」と具体化 |
| 差別化（乱用時） | 何と何の違いかを名指しする。連呼しない |
| 「受け:」等の社内メモ用語 | クライアント文書に社内記法を持ち込まない |

**文体規則**:

- ダッシュ（——／ーー）禁止（原則6・検査基準Bと共通）。読点・句点で区切るか文を分ける
- 疑問形止めの多用禁止。問い立ては必要箇所1〜2回まで。断定文を基本にする
- 体言止めの乱用注意。箇条書き以外の本文では主語＋述語の一文にする
- 「マトリクス」「フレーム」等の分析用語を本文に不用意に出さない。読み手には分析結果だけを見せる
- 不必要な固有名詞・造語を作らない（プロジェクト内呼称・略号の独断発明を含む）

## 原則8: メッセージ・クリスタライズ規範

スライドのメッセージ行・報告書のリード文の品質基準。過去案件で最多の差し戻し領域。

- **1文・単一主張**: メッセージは1文で単一の主張を述べる。要素の羅列・フレームの説明・メタメッセージ（「〜を整理する」「本章の構成」等）は不可。「何も言っていないに等しい」文（当たり前・空疎）を置かない
- **縦読み検査**: タイトル＋メッセージ行だけを上から縦に読み、それだけで文書全体の主張が一本の筋として通るかを確認する。通らなければメッセージを書き直す
- **初見読者テスト**: 略語・固有名詞・記号参照（A-1、Q0 等）は初出で定義するか使わない。資料単体で理解が自己完結すること（原則5と連動）
- **意味重複チェック**: スライド間・章間・箇条書き間で、本質的に同じ主張が重複していないか。重複は統合するか、片方を一段深い主張に差し替える
- **根拠の先回り**: 断定文・トレンド文には出典を先回りで付す。出典のない主張は「出典なし」「仮説」と明示してから提示する（原則4・検査基準Cと連動）
- **示唆の深度**: 「観察→ファインディングス、導出→示唆（仮説）＋なぜそう考えたかを明示」の型で書く。示唆は俯瞰的に踏み込んでよいが、事実を断定しにいく書き方はしない（検査基準A・Cと連動）

# 提出前最終検査基準

原則1〜6が「書くときの指針」であるのに対し、本節は「出す前に走る検査パス」の基準を定める。過去案件で観察された差し戻しの3大クラスタ（①出典・ファクト忠実性 ②断定・言い過ぎトーン ③AI臭・メタ言及）を、提出直前に機械検出＋文脈判定で洗い出すためのもの。quality-reviewer エージェントの提出前最終検査モードが本節を検査基準として使用する。

## 検査基準A: 断定トーン

根拠の強さに見合わない言い切りを検出する。以下の語彙を Grep で機械検出し、文脈判定（その断定を支える根拠が同一セクション内に示されているか）のうえ、①根拠を足す ②表現を一段弱める ③削除する のいずれかで処理する。

**NG語彙リスト（grep 対象）**:

- 断定の格上げ: `収斂`、`最適解`、`決定打`、`唯一の`、`唯一解`、`即決`、`一択`
- 無根拠の強調: `必ず`、`間違いなく`、`疑いなく`、`疑いの余地`、`確実に`、`明らかに`、`断言`
- 押しつけの提言: `すべきである`、`すべきだ`、`べきである`（クライアントが自ら判断すべき領域〔組織再編・体制変更等〕への断定提言は特に不可）
- 選択肢の封殺: `しかない`、`他に選択肢`、`不可避`、`不可欠`

「最適解」「決定打」は AI 典型表現ブロックリスト（E 節のキャッチー単語）にも含まれる。検出後の書き換えは「中立語＋具体根拠」（例: ✕「ハイブリッド検索が最適解となる」→ ○「現時点の公開ベンチマークでは、ハイブリッド検索が精度・コストの両面で優位である（出典URL）」）。迷ったら一段控えめ側に倒す。

## 検査基準B: メタ言及・防御的注記

書き手の作業過程・自信のなさが本文に漏れているパターンを検出する。

- **自己参照の多用**: 「本資料では」「本調査では」「本稿では」が用語定義・出典整理以外で登場していないか（必要最小限＝文書全体で数回まで。段落ごとに現れるのは多用）
- **文書内バージョン・内部参照**: 「（調査提案書v0.2 論点3）」「Phase 1 で分かったことをもとに」のような、作成過程の文書・工程への参照が本文に残っていないか。読み手には無意味な参照であり削除する
- **解釈の二重防御**: 「公開情報では確認できておらず、〜と読める」「〜にあたる（示唆）」のように、留保と解釈を重ねて主張をぼかしていないか。留保が必要なら1回だけ、最小の形で置く
- **同じ留保の反復**: 「公開事例は乏しい」と書いた後に別の箇所で「明示事例は少ない」と再度書く等、同一の留保を2回以上繰り返していないか
- **予告文・案内文**: 「以下で詳しく見ていく」「順を追って整理する」等の案内的表現が残っていないか

**grep 対象**: `本資料では`、`本調査では`、`本稿では`、`と読める`、`（示唆`、`示唆）`、`公開情報では`、`v0.`、`v1.`、`以下で詳し`、`順を追って`

## 検査基準C: 事実／考察（仮説）の区分

本文中の記述が、次の3種のどれであるかを文面上区別できるかを検査する。

| 区分 | 定義 | 表記ルール |
|------|------|-----------|
| 聴取事実 | クライアント・インタビュー対象者から直接聞いた内容 | 「〜と伺った」「ヒアリングでは〜」。**聞いていないことを「伺った」と書かない** |
| 原典事実 | 出典URLで裏付けられる公開情報 | 出典URLを添える（原則4）。URLのないものは事実として書かない |
| 本資料の解釈 | 上記から書き手が導いた仮説・考察・応用 | 「**本資料での解釈・応用**」「**本調査の観察**」等のラベルを明示する |

- 未確認の前提（クライアントの利用ツール・体制等）を既知事実のように書いていないか。確認するか、仮説と明示する
- フレームワークを原典と異なる文脈に適用する場合、「本資料での解釈・応用」ラベルを付けているか（原典超えのラベル付け禁止は検査基準Dと連動）

## 検査基準D: 出典忠実性

引用・参照が原典の射程を超えていないかを、**原典URLの実フェッチ**で突合する。

- **原典にない用語ラベルの貼り付け**: 原典が「データメッシュ」としか言っていない事例を「データファブリックの実装」と位置づける等、原典が使っていない業界用語・フレームワークのラベルを事例に貼っていないか
- **段階・規模の格上げ**: 原典が「PoC 段階」と書いているものを「本番実装」と一般化する等、原典の記述より強い事実に格上げしていないか
