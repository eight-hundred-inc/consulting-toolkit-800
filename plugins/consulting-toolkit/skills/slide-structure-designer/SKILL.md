# スライド構成設計スキル

ソースドキュメントから、スライドデッキの**設計図（骨子）**をMDとして組み立てる。

スライドそのものは描かない。「何を・どの順で・何を主張するか」を決めるのが仕事で、作図・清書・配色・構図決定はしない。構図・図解・グラフの選定は下流のスライド化スキルに委ねる。

## 入出力

- **入力**: ソースドキュメント（議事録・提案書・ディスカッション資料・調査報告等）
- **出力**: スライド構成MD（各スライド＝タイトル・メッセージ・ボディの3層）

## 核心原則

この2つだけは本文に置く。残りの詳細はリファレンスに分離してある。

### 1. 1スライド1メッセージ（3層構造）

各スライドは次の3層で構成する。メッセージが2つ以上になる場合はスライドを分割する。

- **タイトル**: そのスライドが**何についてか**（WHAT）。名詞句で終える。
- **メッセージ**: そのスライドで**何を主張するか**（SO WHAT）。句点で終わる完全文1つ。
- **ボディ**: メッセージを支える根拠・データ・図表。

### 2. タイトルとメッセージの役割分担（WHAT / SO WHAT）

タイトルに結論・主張を埋め込まない。混在させると詳細スライドの見出しとして過剰になる。

| 悪い例（結論をタイトルに） | 良い例（タイトルは名詞句） |
|---|---|
| 許認可は障壁にならない | 許認可要件の比較 |
| OEM選定は使用感コンペで決まる | OEM選定プロセスの構造 |
| 第1章サマリ（技術停滞の構造） | 第1章サマリ |
| 自力参入が困難な理由 — 直近事例は全てM&A | 異業種参入の実例分析 |

## ワークフロー（トップダウン4フェーズ）

上の階層から順に固める。下の階層の手戻りを防ぐため、順序を飛ばさない。

### Phase 1｜デッキの主張を確定する

ソースから次を抽出・確認する。**発明しない**。

1. 読後ゴール（読み手に最終的にどう動いてほしいか）を1文
2. デッキ全体の結論を1文で言い切る
3. 結論を支える根拠を3つ前後に絞る（各根拠が後段で1章になる粒度）
4. 全体を1本の筋に通す：**状況 → 引っかかり（課題）→ 問い → 答え（結論）**

> **上流との境界**: 本スキルの入力は議事録・提案書など**既に構造化されたソース**。ソースに主張がある（提案書・最終報告書等）ならそれを採用する。ソースが分析素材で主張が未確定な場合、主張の創作は上流（`integrated-analysis-creator` / `project-proposal` / `report-outline-creator`）の領分。ここで新規に主張を作らず、「主張が未確定。上流での確定が必要」と質問で返す。

### Phase 2｜章立てとスライドに分解する

1. 根拠を軸に章立て（3〜5章）し、各章のメッセージを1文ずつ書く
2. 各章をスライドに分解し、各スライドのメッセージを1文ずつ書く（1スライド1メッセージ）
3. 情報量が1枚に収まらない章は、構造パターンで分割する → [references/message-logic.md](references/message-logic.md)
   - **ピラミッド構造**（結論→根拠を積み上げる報告系）
   - **問い→回答のカスケード**（読み手と考える提案・ディスカッション系）
4. 全スライドのメッセージ行を順に通読し、重複・飛躍がないかを確認する
5. （カスケードダウン構造を採る報告書の場合）ボディ詳細化の前にL1〜L4の階層整合を検証する → message-logic.md「ピラミッド構造の整合性検証」

初回はやや細かめに設計する。ユーザーの統合・削除指示で粒度を調整する方が、粗く作って後から細かくするより効率的。

### Phase 3｜各スライドを設計する

スライドごとに、テキスト3層を埋める。**1つでも空欄なら未完成**とみなす。

**A. テキスト（3層）**
- タイトル（WHAT・名詞句）／メッセージ（SO WHAT・句点で終わる完全文1つ）
- ボディ：メッセージを支える根拠・データ・図表。全要素がメッセージを補強する方向のみ。診断と処方を1枚に混ぜない → [references/writing-rules.md](references/writing-rules.md)

> **表現の赤旗（本文を書きながら避ける。全リストは [references/quality-checklist.md](references/quality-checklist.md) のF）**
> - **比喩名詞**：弾／核心／突破口／導線／軸足／素地／情報の海 → 機能名詞（中核要因・突破経路・軸となる経験）
> - **物理メタファ動詞**：飲み込まれる／降ろす／突きつける／攻める → 機能動詞（直面する・具体化する・示す）
> - **比喩動詞・俗語**：効く／刺さる／響く／本命／張る／賭ける／一発出し → 主要な・奏功する・採る
> - **ドラマ語・断定キャッチー**：致命的／決定打／最適解／空白地帯／ぽっかり → 中立語＋具体根拠
> - **カタカナ比喩・AI臭**：レッドオーシャン／構造的な帰結／足場／閾値の構造 → 平易な日本語に
> - **ソースのメタファをそのまま転記しない**。意味を保って機能語に書き換える（表のセル内も同じ）

**B. 出典**
- 具体的な数値・事実・事例に `※N` マーカーと末尾出典リスト → writing-rules.md「出典の記載」

### Phase 4｜仕上げ点検

全スライドを設計し終えたら、骨組みはやり直さず、[references/quality-checklist.md](references/quality-checklist.md) の全項目（A〜I）を確認する。違反は修正する。正しく描けていても、**一目で要点が立たなければ未完成**とみなす。

修正を1件受けたら**暗黙の全体ルール**として扱い、同パターンが他スライドに残っていないか走査する（横展開）。発見した修正候補は勝手に書き換えず「他にN件あります。一括修正しますか？」と提案してから着手する。

## 判断基準

### 1枚に収めるか分割するか

- テーブルが7行以下／箇条書きが5項目以下 → 1枚に収まる
- 独立した論点が2つ以上 → 分割を検討
- メッセージを1文で書けない → 分割する
- ユーザーが統合を指示 → 情報量が多くても1枚に収める

### サマリスライドを入れるか

- 詳細スライドが3枚以上 → サマリを入れる
- 詳細スライドが2枚以下／結論が自明 → 省略可

## 下流スキルとの連携（責務の高さで分担）

本スキルは「**何を主張するか**（タイトル・メッセージ・ボディの3層）」までを決める。「**どう描くか**」（並べ方・図解/グラフの選定・軸・配色・フォント・実レンダリング・ノード数・座標）は下流に委ねる。

- **図解（概念フロー・2x2・ピラミッド・ファネル・サイクル・ベン図・組織図・レイヤー積層・スイムレーン等）** → `html-artifact/references/diagram-components.md`
- **データチャート（棒・積み上げ・レンジ・レーダー・2パネル等）** → `chart-generator-guide`

スライド化は次の3段階で進める：

1. **① スライド構成MD（本スキルの出力）** — `Output/スライド構成.md`
2. **② HTMLスライド** — `html-artifact`（Slide Deck format）で16:9 HTMLデッキに変換
3. **③ PPTX** — ②のHTMLを「デザイン見本」に、ブランドpptxスキルでネイティブPPTX化

②を介さず構成MDから直接③に渡してもよい。
