# 構図の決め方（メッセージ → どう見せるか）

各スライドのメッセージを決めたら、そのメッセージが一目で立つ構図を**同時に**決める。本ファイルは「どの構図か」を選ぶための参照メニュー。

**「どう描くか」は複製しない。** 図解・グラフのカタログ本体は下流が持つので、本スキルでは**名前で指名する**だけにする（下流＝`html-artifact/references/diagram-components.md`、`chart-generator-guide`）。ここで別カタログを作ると、設計と描画で名前がズレて受け渡しが壊れる。

---

## 1. 軸を探す5つの切り口

情報を整理する軸は、次の5つから探す。手薄な所（抜け）も指摘する。

**種類／時間／場所／順番／大小**

同じ語が複数の項目で繰り返されているなら、その語は軸に括れるサイン。1枚に詰める軸は**2つまで**。3つ目は「図を分ける／左右に割る／一部を拡大」のいずれかで逃がす。

---

## 2. 並べ方6型（メッセージ → 構造）

メッセージの主張の型から、最も伝わる並べ方を1つ選ぶ。各型がどの下流コンポーネントで実現されるかを併記する（指名はこの名前で行う）。

| 型 | 使う場面 | 構造 | 下流での実現（指名先） |
|---|---|---|---|
| **順番** | 工程・時間の流れ | 左→右の矢印で並べる | Concept Flow（横）／ステップフロー型／タイムライン・ロードマップ型 |
| **対比** | A案/B案、現状/理想 | 左右に同じ粒度で並べる | State Grid（2列 As-Is/To-Be）／2パネル横棒（chart）／マトリクス型 |
| **仕分け** | 種類でMECEに分ける | 表またはツリー | Quadrant Matrix（2x2）／Org Tree（ツリー）／Layer Stack（層） |
| **問題と打ち手** | 課題→施策→効果 | 問題→打ち手→効果の3ブロック | Concept Flow（横3ノード）／作り込み図版 |
| **原因と結果** | なぜ→だから | 因果で筋を通す | Concept Flow（横・辺ラベル付き） |
| **結論先出し** | 結論＋根拠 | 結論を上、根拠を下に置く三角 | Pyramid／ピラミッド構造（message-logic.md） |

選んだ型は、各スライド設計に「並べ方型＋理由1行」で記録する。

---

## 3. ボディの構造はメッセージの論理構造を反映させる

ボディの内容がメッセージに関連していても、**構造**（表のカラム設計・行のグルーピング・図解の選択）がメッセージの**論理構造**と噛み合っていなければ、読み手は表を眺めた後に自力で結論を導く羽目になる。メッセージの主張が構造から自然に読み取れる状態を作る。

### 原則1：対比のメッセージには対比の構造

メッセージが「AとBを比較するとAの方がXである」なら、表も対比構造にする。

- 良い例：メッセージ「効率化と格上げを同時に進めている」→ カラムを「効率化の施策 / 格上げの施策」にし、各行（企業）が両方やっていると一目でわかる
- 悪い例：カラムが「課題 / 施策 / ステージ / 対応」→ 全行読んでから読み手が自分で「同時にやっている」と気づく必要がある

### 原則2：1つのメッセージに1つの視覚焦点

1枚に複数の表・図がある場合、すべてが同一メッセージを支える構造になっていなければならない。異なる論点を支える2つの図がある場合は、スライドを分割するか、一方を他方に統合する。

### 原則3：詳細度はメッセージの抽象度と揃える

メッセージがグルーピングレベル（「前半と後半で効果が異なる」）なのに、ボディが個別レベル（8工程の等重み一覧）だと、論点が表に埋もれる。まずメッセージの抽象度に合ったグルーピングで提示し、個別詳細はAppendixかブロック内に収める。判定基準：**この図を見てメッセージの主張が3秒で読み取れるか**。

## 4. 図解の指名（→ diagram-components.md）

概念を視覚化する場合は、下流の図解コンポーネントを名前で指名する。**自前で図解を発明・再カタログ化しない。**

| 表現したい内容 | 指名するコンポーネント |
|---|---|
| 段階的な流れ・プロセス | Concept Flow（横/縦） |
| 2軸での分類・優先度判断 | Quadrant Matrix（2x2） |
| 階層・優先度・抽象度の構造 | Pyramid |
| 絞り込み・歩留まり | Funnel |
| 繰り返すループ・改善サイクル | Cycle |
| 集合・重なり・共通領域 | Venn |
| 組織・意思決定階層・分類ツリー | Org Tree |
| 技術スタック・責任分担・層 | Layer Stack |
| 横＝進行、縦＝部署/会議体の流れ | スイムレーン型 |
| 縦＝取り組み、横＝時間 | タイムライン・ロードマップ型 |

選択の詳細・ノード数の上限・「リッチ判定」「型に当てはめて情報を落とさない」原則は diagram-components.md 側にある。本スキルは「このスライドは Funnel」「これは Quadrant」と指名するところまでで、ノード数や座標は下流が決める。

**標準8図解で形にならない固有構造**（業界地図・商流物流の二層図・全体＋拡大など）は、「作り込み図版（per-figure）が必要」と注記して下流に渡す。

## 5. グラフの指名（→ chart-generator-guide）

数字を扱うスライドには、見せたい意図からグラフを1つ指定する。下流の標準チャートカタログにある型を名前で指名する。

| 見せたい意図 | 指名するチャート（標準カタログ） |
|---|---|
| 差・ランキング・度数比較 | 横棒（単系列・先頭ハイライト） |
| グループ間の差 | 横棒（2系列オフセット） |
| 3系列以上の横断比較 | 縦棒（複数系列） |
| 構成比・Likert尺度 | 横棒（積み上げ） |
| 区間・レンジ（年収帯等） | 横棒（レンジ） |
| 多軸プロフィール比較（4〜6軸） | レーダー |
| Before/After・優先順位の変化 | 2パネル横棒 |

**標準カタログにない型**（推移＝折れ線、増減分解＝滝、相関＝散布、規模×割合＝面積、重複＝ベン）は、無理に流用せず「標準外 → matplotlibカスタム or 作り込み図版」と注記して下流に渡す（重複＝ベン、歩留まり＝ファネルは §4 の図解として扱う）。

### グラフ共通の規律（設計段階で決めること）

- **主役は1本/1系列だけ強調**、他はグレー。強調はメッセージの主張と連動させる
- 棒グラフの軸は原則0始まり
- 推移以外（カテゴリ比較）に折れ線を使わない
- 実績＝実線、見込み＝破線で分ける
- 系列は4本まで

レンダリング上の細則（色トークン・凡例位置・フォント）は chart-generator-guide 側。

## 6. 強調・軸・視線（設計段階の視覚規律）

「どう描くか」は下流だが、次の**設計判断**は構成段階で決めて記録する。

- **主役の1点を絞る**：1スライドで一番見せたい数字・差・転換点を1つだけ指定（色／吹き出し／網かけのいずれか）。脇役はトーンを落とす
- **軸は2つまで**：3つ目は図を分ける／左右に割る／一部を拡大
- **視線はZ字**：最重要を左上、結論を右下に置く配置を想定する
- **白黒で構造が成立するか**：色に頼らず、濃淡・罫線・配置・ラベルで構造が読めるか。色は最後（下流が単色アクセントで付ける）
- **事実と解釈の分離**：見出し＝事実、図＝根拠、解釈（So What）＝別枠（右か下）

---

## チェック（構図の点検）

- 並べ方型が当たっていて、メッセージの主張がその構造から3秒で読み取れるか
- 図解・グラフを下流カタログの名前で指名しているか（自前カタログを作っていないか）
- 標準カタログ外の図解/グラフを「作り込み図版が必要」と注記しているか
- 強調する1点が決まっているか。脇役が主役を食っていないか
- 1枚に詰めた軸が2つ以内か
