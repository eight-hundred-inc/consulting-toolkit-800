# プロジェクト提案書作成スキル（調査型／汎用型）

## タイプ判定（最初に必ず行う）

リクエスト内容・与件から、以下のどちらのタイプかを判定し、以降の構成・ルールを切り替える。

- **調査型**：デスクリサーチ＋デプスインタビューを主体とする調査プロジェクト（市場調査、競合調査、技術動向調査、デューデリジェンス等）。「調査提案書」「リサーチ提案書」等の依頼、または与件が「〜を調べる／検証する／情報が不足している」系のとき。
- **汎用型**：デスクリサーチ＋インタビューを主体としないプロジェクト（戦略策定、コンテンツ制作、システム実装、事業計画等）。「プロジェクト提案書」「提案書を作って」等の依頼で、成果物が分析・設計・実装・実行計画に及ぶとき。

判定に迷う場合は、**「主たる作業がデスクリサーチ＋インタビューによる情報収集・検証か（→調査型）」「分析・設計・実装・実行計画が主か（→汎用型）」**で切り分ける。判定が割れる場合はユーザーに確認する。

**デプスインタビュー前提の設計条件（調査型のみ）**：調査型のうち、インプットや指示から**デプスインタビューが必要と推測できる場合のみ**、エキスパート／潜在顧客インタビューを前提とした設計（インタビュー対象の属性明記、設問設計の依存関係等）を含める。デスクリサーチのみで足りる調査型では、インタビュー前提の記述は入れない。

## 入出力

- **入力（必須）**: 与件情報（背景、プロジェクト／調査テーマ、予算・期間）、クライアント打ち合わせメモ（優先度、具体的要望）
- **入力（任意）**: 論点・仮説.md（検証済み）、調査レポート、既存資料
- **出力**: 提案書.md（スライド骨子形式）

## 重要：本スキルが狙う「内容品質」

このスキルのゴールは、型を埋めることではなく **読み手の意思決定が進む提案内容** を出すこと。

**このスキルを使うときは必ず**以下を読み、指示を守ること（例外なし）：
- `references/quality-playbook.md`

## 提案書の構成

タイプによって「アプローチ／タスク」「スコープ／調査対象」などの節名と有無が異なる。以下の対応で組む。

### 汎用型の構成

0. 支援実績の伝え方（任意） - 面談前に先方へ共有する匿名化した実績サマリ
1. 提案概要 - プロジェクトの観点と「何を実現し、どのような意思決定を支えるか」の結論
2. 背景・目的 - クライアントの課題と意思決定に足りないもの（情報・分析・設計・実装等）
3. スコープ - 対象範囲・除外事項・前提条件
4. 論点 - 大論点と小論点のイシューツリー形式（プロジェクトゴールに直結）
5. 初期仮説 - 論点に紐づく、数字と根拠を入れた仮説
6. アプローチ概要 - 論点と対応したアプローチ（結論に至る道筋）
7. アプローチ詳細 - 論点対応・依存関係・具体作業・成果物を含む詳細
8. 納品物 - 会議転用可能な成果物一式の定義
9. 体制 - プロジェクト体制・役割分担
10. スケジュール - 週次で分解し、中間報告→最終収束の設計
11. お見積り - プランごとにカバーする論点・アプローチ範囲が異なり、それに応じた見積りを提示

### 調査型の構成

1. 提案概要 - 調査の観点と「何を絞り込むか」の結論
2. 背景・目的 - クライアントの課題と会議意思決定に必要な不足
3. 調査対象 - 技術/プレイヤー/供給構造のスコープ
4. 論点 - 大論点と小論点のイシューツリー形式（事業設計に直結）
5. 初期仮説 - 論点に紐づく、数字と根拠を入れた仮説
6. タスク概要 - 論点と対応した調査タスク（結論に至る道筋）
7. タスク詳細 - 論点対応・依存関係・参照ソース例示を含む具体作業
8. 納品物 - 会議転用可能なPPT章立て＋一覧（Excel等）
9. スケジュール - 週次で分解し、中間報告→最終収束の設計
10. お見積り - プランごとにカバーする論点・タスク範囲が異なり、それに応じた見積りを提示
11. 体制・進め方（補足） - 学習反映の仕組み、リスク管理
12. 次アクション - 3者面談で確定すべき意思決定事項

### 節名の対応

| 概念 | 汎用型 | 調査型 |
| --- | --- | --- |
| 実行単位 | アプローチ（分析・設計・実装・レビュー等） | タスク（デスクリサーチ・インタビュー・分析統合等） |
| 対象範囲 | スコープ（対象範囲・除外・前提） | 調査対象（技術/プレイヤー/供給構造） |
| 実績提示 | 0. 支援実績の伝え方（任意） | （原則入れない。必要時は面談で口頭補足） |
| 面談確定事項 | （体制・スケジュールに内包） | 12. 次アクション（3者面談で確定する事項） |

## ワークフロー

1. **タイプ判定** → 上記「タイプ判定」で調査型／汎用型を確定する
2. **前提の抽出** → クライアント名、対象範囲（調査型は対象地域・対象）、除外事項、納期（中間/最終）、予算レンジ、社内会議用途を先に固定
3. **支援実績の整理（汎用型のみ・任意）** → 面談前に先方へ共有する匿名化した実績サマリを準備（類似案件の経験、アウトプットの型）
4. **提案概要の確定** → 観点（何を、どの切り口で取り組む／調べるか）を明示し、「何を実現する／絞り込む提案か」を一文で言い切る
5. **論点設計** → `Output/論点・仮説.md` が存在する場合はそれをベースにする（ゼロから設計しない）。存在しない場合は大論点3〜5個、各大論点に小論点2〜4個を「問い形式」で設定（プロジェクトゴール／事業設計に直結させる）
6. **初期仮説の構築** → 既存資料や調査レポートがある場合は、得られた数字・固有名詞・根拠を直接反映する。論点に直接紐づけ、数字と根拠を入れた仮説にする（レンジ、コスト、プレミアム額、要因の因数分解など）
7. **アプローチ／タスク設計** → 既存情報や不足情報をもとに具体化する。
   - 汎用型：論点ごとにアプローチ手法（分析・設計・実装・レビュー等）を割り当て、具体的な作業内容を記述する
   - 調査型：デスクリサーチ＋（必要なら）エキスパート/潜在顧客インタビューを論点に割り当て、参照ソース例を具体的に列挙する
8. **納品物の具体化** → 最終成果物の章立て・構成、比較軸、会議転用（中間報告の最低ライン）を明記。調査型は一覧（企業/大学/OEM/評価等）も明記
9. **体制設計（汎用型）／進め方の補足（調査型）** → 汎用型はプロジェクト体制・役割分担・クライアント側協力体制を明確化。調査型は学習反映の仕組み・リスク管理を補足で示す
10. **見積り作成** → 論点・アプローチ／タスクのカバー範囲で差分を設計し、範囲に応じた見積りを3プランで提示する
11. **スケジュール設計** → 週次で分解し、各週のアプローチ／タスクと中間共有のタイミングを明確化
12. **次アクションの明記（調査型のみ）** → 3者面談で確定すべき意思決定事項（論点優先度、対象定義、成功条件等）を列挙する
13. **品質ゲート** → `references/quality-playbook.md` の「品質バー」と「禁止ワード」を満たさない箇所を削る/書き直す

## 内容品質ルール（常時適用）

### 共通ライティング原則（全セクションに適用）

- 読み手はクライアント自身。「貴社」で書く。作業プロセスへの言及は削除
- 曖昧な定性表現ではなく具体的な数字・固有名詞・時間軸で記述
- すべての記述が「So What?」に答えている。整理・把握で終わらない
- ソースにない数字・事実を捏造しない。不明は「不明」と明記
- 略語は初出展開。分類・専門用語はその場で定義。結論文に主語を明記
- 本文は「読み手の意思決定に必要なもの」だけで構成する。作成プロセスのメタ言及・既往実績への自己言及・営業章/営業フレーズ・スコープ外の長い羅列・口語/煽り/AI臭表現は排除する
- **起草時に `${CLAUDE_PLUGIN_ROOT}/skills/_shared/writing-principles.md` の原則7（NG語彙・言い換え辞書）・原則8（メッセージ・クリスタライズ規範）を参照する**。辞書のNG語彙を grep で潰し、各セクションのリード文は単一主張の1文に収める

### 提案書固有のルール（共通）

- **固有名詞を入れる**（プレイヤー群、比較ベンチマーク、工程、技術要素、規制カテゴリなど）
- **「何が壁か」を因数分解し、突破口（組むべき相手/必要投資/必要根拠）まで一歩踏み込む**

### 調査型固有のルール（調査型のときのみ適用）

- **事前送付前提の粒度**：提案書は3者面談前の事前送付を想定し、 **「これから調査する立場」** で書く。事前デスクリサーチで得た詳細情報（網羅的な企業/研究室リスト、特定固有名詞、確定数値）を提案書に網羅的に出さない。詳細は内部成果物（`Output/論点・仮説.md` ／ `プロジェクトサマリ.md`）に残し、提案書には例示レベル（「例：A／B／C 等」）で触れる
- **作業プロセスラベルを出さない**：「実施済」「検証済み」「棄却」「修正」など、内部成果物で使う作業状態ラベルを提案書に出さない。仮説は「〜と想定される」「〜可能性が高い」「〜と考えられる」で統一し、確定情報の提示は中間報告以降に取る
- **職人スラング・口語表現を避ける**：「網を張る」「やり込む」「ぶっ刺さる」等の表現は、文章の品位を下げ初見の読み手の認知負荷になる。「対象とする」「構成する」「直結させる」等の中立的な動詞に置換

## セクション別の形式ルール

### 論点形式（共通）

- 大論点・小論点ともに**問い形式**で記載
- 汎用型の例：「現行システムのどの機能領域に最も改善余地があり、ROI最大化に寄与するか？」
- 調査型の例：「カテゴリ（医薬品／部外品／化粧品）別に、必要エビデンスと表現可能範囲は何か？」

### 初期仮説形式（共通）

- **数字を入れる**：レンジ、コスト、効果、プレミアム額、比率などを具体的に示す
- **要因を因数分解する**：「〜可能性が高い」だけでなく、なぜそうなるかの構造を示す
- **論点（大論点→小論点）に直接紐づく階層**で記載する
- 詳細とテンプレートは [references/templates.md](references/templates.md) の「初期仮説」を参照

### アプローチ詳細形式（汎用型）

- アプローチごとに**対応する論点（小論点番号）**を明記する
- アプローチ間の**依存関係**（前工程→後工程）を明示する
- **具体的な手法・フレームワーク・ツール**を記載する（抽象的な「分析する」だけで終わらない）
- 良い例・悪い例は [references/templates.md](references/templates.md) の「アプローチ詳細」を参照

### タスク詳細形式（調査型）

- タスクごとに**対応する論点（小論点番号）**を明記する
- タスク間の**依存関係**（前工程→後工程）を明示する（例：デスクリサーチの結果がインタビュー設問設計の前提になる）
- **参照ソースを具体的に列挙**する（抽象的な「公開情報」だけで終わらない）
- **インタビュー対象の属性**を明記する（例：転職エージェント、競合OB、潜在顧客など）
- **「提示値」と「実態値」の乖離**に着目する視点を入れる
- 良い例・悪い例は [references/templates.md](references/templates.md) の「タスク詳細」を参照

### 見積り形式（共通）

- プランごとに**カバーする論点・アプローチ／タスクの範囲**が異なり、範囲に応じて見積金額が変わる構造にする
- 金額は具体的に記載する（「予算上限内で調整」だけで終わらない）
- 調査型では、対象者母数が不足する場合の方針（段階拡張、提示値の併用等）を全プラン共通で併記する
- テンプレートは [references/templates.md](references/templates.md) の「見積り」を参照

### 納品物形式（共通）

- シナリオ比較表（選択肢A/B/Cの費用対効果）を含める
- テンプレートは [references/templates.md](references/templates.md) の「納品物」を参照

### スケジュール形式（共通）

- **週次で分解**する（月単位の大雑把なスケジュールは禁止）
- 各週のアプローチ／タスクと中間共有のタイミングを明確化
- テンプレートは [references/templates.md](references/templates.md) の「スケジュール」を参照

### 次アクション形式（調査型）

- 3者面談で確定すべき意思決定事項（論点優先度、対象定義、推定ロジックの合意、中間報告の最低ライン、第1フェーズの成功条件等）を列挙する
- テンプレートは [references/templates.md](references/templates.md) の「次アクション」を参照
