## スライド形式（共通）

- **メッセージ**: そのセクションの結論・打ち手・So what を一文で書く。読み手が得る価値や判断材料が伝わるようにする。

```markdown
## セクション番号. セクションタイトル

結論・打ち手を40〜50文字程度で。体言止め不可、です・ます調不可

* **項目A**：内容
  * 詳細a1
  * 詳細a2
```

## 背景・目的

- 配下で「背景」「現状課題」「目的」「除外」を必ず分けて記載
- 「現状課題」には、意思決定に足りないものを具体的に書く

```markdown
## 1. 背景・目的

〇〇の意思決定に使える△△を短期で設計する

* **背景**：...
* **現状課題**：方針は決定済みだが、**具体的なシナリオ比較・投資対効果の試算・実現性の評価**が不足しており、経営判断に進めない
* **目的**：...
* **除外**：...
```

## 初期仮説

### 形式ルール

- **完備な文章**で記載（体言止め・断片的箇条書き禁止）
- 述語：「〜と考えられる」「〜と想定される」「〜可能性が高い」等
- 主語を明確に（例：「A社は〜」「現行システムは〜」）
- **仮説は二層**で書く：上位仮説（構造）→下位仮説（具体）
- **論点（大論点→小論点）に直接紐づく階層**で記載する

### 数字と根拠を入れる（重要）

- 仮説には**具体的な数字**を入れる（コスト、削減率、期間、比率など）
- 数字が入れられない場合でも、**要因の因数分解**を示す
- 「〜可能性が高い」だけの羅列は禁止

**良い例**：
- 「受発注業務の手作業比率は60〜70%と推定され、自動化により年間XX百万円の工数削減が見込まれる」
- 「導入障壁は"既存システム連携"と"現場オペレーション変更"の同時対応が必要な点に集約される」
- 「フェーズ1の投資回収は12〜18ヶ月、フェーズ2は24〜30ヶ月と想定される」

**悪い例**：
- 「コスト削減効果がある可能性が高い」（数字がない）
- 「導入障壁が存在する可能性が高い」（要因の分解がない）

### テンプレート

```markdown
## 4. 初期仮説

要因を因数分解し、打ち手の優先順位を付けられる形に落とす

* **大論点① 仮説：** 〇〇領域の自動化により年間XX〜XX百万円の効果が見込まれ、投資回収は12〜18ヶ月と想定される
  * 小論点①-1 仮説：現行業務の手作業比率はXX〜XX%であり、△△工程が最もボトルネックになっていると考えられる
  * 小論点①-2 仮説：段階導入（Phase1: △△、Phase2: □□）により、導入リスクを最小化しつつ早期に効果を刈り取れると想定される
```

## アプローチ概要

- 論点との対応関係を表形式で明示
- アプローチ間の依存関係（前提アプローチ）を明示し、実行順序がわかるようにする

```markdown
| アプローチ | 対応論点（小論点） | 前提 | 内容 |
| --- | --- | --- | --- |
| A1: 現状分析 | ①-1 / ①-2 | — | ... |
| A2: ベンチマーク分析 | ②-1 / ②-2 | A1 | ... |
| A3: シナリオ設計 | ③-1 / ③-2 | A1, A2 | ... |
| A4: 実行計画策定 | ④-1 / ④-2 | A1, A2, A3 | ... |
```

---

## アプローチ詳細

### 形式ルール

- **具体的な手法・フレームワーク・ツール**を記載する（抽象的な「分析する」だけで終わらない）
- **アウトプットの形**を明記する（表、マトリクス、フロー図、比較表など）

### 良い例

```markdown
* **A1: 現状分析**（対応論点：①-1 / ①-2｜前提：なし）
  * 業務フロー可視化: 対象3部門（受発注・在庫管理・出荷）の業務フローをヒアリング＋現地観察で可視化
    * 各工程の手作業/自動化比率、処理時間、エラー率を定量化
    * ボトルネック工程をコスト×頻度のマトリクスで優先順位付け
  * システム構成の棚卸し: 現行システム（ERP/WMS/TMS）の機能マップと連携状況を整理
    * API連携可否、データフォーマット互換性、カスタマイズ範囲を一覧化
  * アウトプット: 業務フロー図（As-Is）、ボトルネック優先順位マトリクス、システム連携図

* **A2: ベンチマーク分析**（対応論点：②-1 / ②-2｜前提：A1）
  * 同業他社のDX先行事例を5〜8社収集し、導入技術・効果・期間を比較表に整理
  * 先行事例から成功パターン（段階導入 vs 一括導入、内製 vs 外注）を抽出
  * A1の現状分析結果と照合し、貴社に適用可能な手法を選別
  * アウトプット: ベンチマーク比較表、適用可能手法の推奨リスト
```

### 悪い例

```markdown
* 現状分析：現状を分析・整理する
* ベンチマーク：他社事例を調べる
```

## 納品物

- シナリオ比較表（選択肢A/B/Cの費用対効果）を含める

```markdown
## 8. 納品物

〇〇と△△を経営会議で即決できる資料一式として短期間で整える

* **最終報告書（PowerPoint形式）**
  * 現状分析結果（業務フロー・ボトルネック・コスト構造）
  * ベンチマーク比較と推奨アプローチ
  * 選択肢A/B/Cの**シナリオ比較（投資額・効果・リスク・実現期間）**
* **データブック（Excel形式）**
  * 業務量・コスト集計データ
  * ベンチマーク比較一覧
  * ROI試算モデル（前提変数を変更可能）
* **実行ロードマップ**
  * フェーズ別の実行計画（マイルストーン・担当・成果物）
```

## 体制

```markdown
## 9. 体制

貴社側と弊社側の役割を明確に分け、意思決定のスピードを確保する

* **弊社側体制**
  * プロジェクトリード: 1名（全体統括・クライアント窓口）
  * コンサルタント: 1〜2名（分析・設計・資料作成）
  * 専門アドバイザー: 必要に応じて（技術/業界知見）
* **貴社側体制（お願い事項）**
  * プロジェクトオーナー: 1名（最終意思決定権限）
  * 実務担当者: 各部門1名（ヒアリング・データ提供・レビュー）
* **コミュニケーション設計**
  * 週次進捗共有（30分、オンライン）
  * 中間報告会（対面推奨）
  * Slack/Teams等での随時連絡
```

## スケジュール

- **週次で分解**する（月単位の大雑把なスケジュールは禁止）
- 各週のアプローチと中間共有のタイミングを明確化

```markdown
## 10. スケジュール

設計・分析・検証を週次で回し、中間共有を挟んで〇月末に経営判断用最終報告へ確実に着地する

* **W1（○月上旬〜中旬）**
  * キックオフ、対象範囲確定、情報提供依頼
  * ヒアリングスケジュール確定
* **W2〜W3（○月中旬〜下旬）**
  * 現状分析（業務フロー可視化、システム棚卸し）
  * ベンチマーク収集開始
* **W4（○月上旬）**
  * ベンチマーク分析完了
  * **中間共有**（現状分析結果、暫定示唆、シナリオ方向性）
* **W5〜W6（○月中旬）**
  * シナリオ設計、ROI試算、実行計画策定
  * シナリオ比較表のたたき台作成
* **W7〜W8（○月下旬〜納期）**
  * 最終資料化、実行ロードマップ確定
  * **納品**
```

## 見積り

- プランごとに**カバーする論点・アプローチの範囲**が異なり、範囲に応じて見積金額が変わる構造にする
- 金額は具体的に記載する（「予算上限内で調整」だけで終わらない）

| プラン | 価格（税抜） | カバー論点 | 含むアプローチ | 想定アウトプット |
| --- | ---: | --- | --- | --- |
| A：ライト | 180万円 | 大論点①②（骨格） | A1: 現状分析＋A2: ベンチマーク | 基本レポート |
| B：スタンダード（推奨） | 280万円 | 大論点①②③ | A＋A3: シナリオ設計＋比較表 | レポート一式＋比較表 |
| C：プレミアム | 380万円 | 全論点（①〜④） | B＋A4: 実行計画＋ロードマップ | 詳細データブック＋実行ロードマップ |

## 支援実績の伝え方

- 面談前に先方へ共有する**匿名化した実績サマリ**を準備
- 類似案件の経験、アウトプットの型を示し、信頼を構築

```markdown
## 0. 支援実績の伝え方（匿名化）

〇〇と△△を組み合わせ、意思決定資料まで一気通貫で作る

* **先方共有用の実績サマリ文案（固有名詞・数値は伏せる前提）**
  * 〇〇領域で、△△を分析・設計し、経営層決裁用の根拠資料に落とし込む支援を実施
  * 現状分析とベンチマーク比較を突合し、複数シナリオの費用対効果を定量比較する設計を採用
  * 実現リスクに備え、段階的導入計画と代替案を準備
* **アウトプットの型（過去事例で多用する型）**
  * 「現状分析×ベンチマーク×シナリオ比較」の三点セットで、選択肢A/B/Cを比較可能にする
  * フェーズ別の実行計画と、必要投資（工数・費用）を並列表にする
```
