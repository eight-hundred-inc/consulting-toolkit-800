# セクション別テンプレート

提案書の各セクションで使用するテンプレート集。必要なセクションのみ参照すること。

タイプ（調査型／汎用型）で節名・内容が異なる箇所は、それぞれ「【汎用型】」「【調査型】」で分けて示す。共通の節はタイプに関わらず使用する。

## スライド形式（共通）

- **メッセージ**: そのセクションの結論・打ち手・So what を一文で書く。読み手が得る価値や判断材料が伝わるようにする。

```markdown
## セクション番号. セクションタイトル

結論・打ち手を40〜50文字程度で。体言止め不可、です・ます調不可

* **項目A**：内容
  * 詳細a1
  * 詳細a2
```

## 背景・目的（共通）

- 配下で「背景」「現状課題」「目的」を必ず分けて記載（汎用型は加えて「除外」も）
- 「現状課題」には、意思決定に足りないもの／既存情報の不足点を具体的に書く

**【汎用型】の例**

```markdown
## 1. 背景・目的

〇〇の意思決定に使える△△を短期で設計する

* **背景**：...
* **現状課題**：方針は決定済みだが、**具体的なシナリオ比較・投資対効果の試算・実現性の評価**が不足しており、経営判断に進めない
* **目的**：...
* **除外**：...
```

**【調査型】の例**

```markdown
## 1. 背景・目的

〇〇の意思決定に使える△△を短期で提示する

* **背景**：...
* **現状課題**：既存レポート（例：矢野「〇〇」等）は市場概況中心で、**先端技術・技術課題・プレイヤー**の深掘りが不足
* **目的**：...
```

## 初期仮説（共通）

### 形式ルール

- **完備な文章**で記載（体言止め・断片的箇条書き禁止）
- 述語：「〜と考えられる」「〜と想定される」「〜可能性が高い」等
- 主語を明確に（例：「A社は〜」「現行システムは〜」）
- **仮説は二層**で書く：上位仮説（構造）→下位仮説（具体）
- **論点（大論点→小論点）に直接紐づく階層**で記載する

### 数字と根拠を入れる（重要）

- 仮説には**具体的な数字**を入れる（コスト、削減率、期間、レンジ、プレミアム額、比率など）
- 数字が入れられない場合でも、**要因の因数分解**を示す
- 「〜可能性が高い」だけの羅列は禁止

**良い例**：
- 「受発注業務の手作業比率は60〜70%と推定され、自動化により年間XX百万円の工数削減が見込まれる」
- 「導入障壁は"既存システム連携"と"現場オペレーション変更"の同時対応が必要な点に集約される」
- 「市場の中心レンジは、PMが1,000〜1,400万円、PLが800〜1,200万円に収斂しやすい」
- 「高騰は"需給逼迫"と"上流化"が同時に進行した結果であると考えられる」

**悪い例**：
- 「コスト削減効果がある可能性が高い」（数字がない）
- 「乖離が生じている可能性が高い」（要因の分解がない）

### テンプレート

```markdown
## 4. 初期仮説

要因を因数分解し、打ち手の優先順位を付けられる形に落とす

* **大論点① 仮説：** 〇〇領域の自動化により年間XX〜XX百万円の効果が見込まれ、投資回収は12〜18ヶ月と想定される
  * 小論点①-1 仮説：現行業務の手作業比率はXX〜XX%であり、△△工程が最もボトルネックになっていると考えられる
  * 小論点①-2 仮説：段階導入（Phase1: △△、Phase2: □□）により、導入リスクを最小化しつつ早期に効果を刈り取れると想定される
```

## アプローチ概要【汎用型】

- 論点との対応関係を表形式で明示
- アプローチ間の依存関係（前提アプローチ）を明示し、実行順序がわかるようにする

```markdown
| アプローチ | 対応論点（小論点） | 前提 | 内容 |
| --- | --- | --- | --- |
| A1: 現状分析 | ①-1 / ①-2 | — | ... |
| A2: ベンチマーク分析 | ②-1 / ②-2 | A1 | ... |
| A3: シナリオ設計 | ③-1 / ③-2 | A1, A2 | ... |
| A4: 実行計画策定 | ④-1 / ④-2 | A1, A2, A3 | ... |
```

---

## アプローチ詳細【汎用型】

### 形式ルール

- **具体的な手法・フレームワーク・ツール**を記載する（抽象的な「分析する」だけで終わらない）
- **アウトプットの形**を明記する（表、マトリクス、フロー図、比較表など）

### 良い例

```markdown
* **A1: 現状分析**（対応論点：①-1 / ①-2｜前提：なし）
  * 業務フロー可視化: 対象3部門（受発注・在庫管理・出荷）の業務フローをヒアリング＋現地観察で可視化
    * 各工程の手作業/自動化比率、処理時間、エラー率を定量化
    * ボトルネック工程をコスト×頻度のマトリクスで優先順位付け
  * システム構成の棚卸し: 現行システム（ERP/WMS/TMS）の機能マップと連携状況を整理
    * API連携可否、データフォーマット互換性、カスタマイズ範囲を一覧化
  * アウトプット: 業務フロー図（As-Is）、ボトルネック優先順位マトリクス、システム連携図

* **A2: ベンチマーク分析**（対応論点：②-1 / ②-2｜前提：A1）
  * 同業他社のDX先行事例を5〜8社収集し、導入技術・効果・期間を比較表に整理
  * 先行事例から成功パターン（段階導入 vs 一括導入、内製 vs 外注）を抽出
  * A1の現状分析結果と照合し、貴社に適用可能な手法を選別
  * アウトプット: ベンチマーク比較表、適用可能手法の推奨リスト
```

### 悪い例

```markdown
* 現状分析：現状を分析・整理する
* ベンチマーク：他社事例を調べる
```

## タスク概要【調査型】

- 論点との対応関係を表形式で明示
- タスク間の依存関係（前提タスク）を明示し、実行順序がわかるようにする
- 調査手法：デスクリサーチ、エキスパートインタビュー、潜在顧客インタビュー

```markdown
| タスク | 対応論点（小論点） | 前提タスク | 内容 |
| --- | --- | --- | --- |
| T1: デスクリサーチ | ①-1 / ①-2 | — | ... |
| T2: エキスパートインタビュー | ①-3 / ③-1 | T1 | ... |
| T3: 潜在顧客インタビュー | ③-2 | T1 | ... |
| T4: 分析・統合 | ④-1 / ④-2 | T1, T2, T3 | ... |
```

---

## タスク詳細【調査型】

### 形式ルール

- **参照ソースを具体的に列挙**する（抽象的な「公開情報」だけで終わらない）
- **インタビュー対象の属性**を明記する（例：転職エージェント、競合OB、潜在顧客など）
- **「提示値」と「実態値」の乖離**に着目する視点を入れる

### 良い例

```markdown
* **T1: デスクリサーチ**（対応論点：①-1 / ①-2 / ②-1｜前提タスク：なし）
  * 参照候補ソース例（Phase1で必ず当たり、整合性を取る）
    * タレントリンク：ERPコンサル年収（紹介実績ベース）
    * ワークサーチ：給与統計（自己申告/求人由来）
    * キャリアフォワード：フリーランス調査（単価の参照点として利用）
  * ベンチマーク企業群の求人票を収集し、年収レンジ/必須要件/訴求軸/勤務地・働き方を構造化
* **T2: エキスパートインタビュー**（対応論点：①-3 / ②-2 / ③-1｜前提タスク：T1）
  * 対象：ERP特化エージェント/ヘッドハンター
  * 聴取内容：直近12〜24か月の決定年収レンジ実例、上振れ要因、採用チャネル別の勝ちパターン
  * 「提示年収」と「決定年収」の乖離がどこで生じるかを聴取
  * T1のデスクリサーチ結果をもとにインタビュー設問を設計
```

### 悪い例

```markdown
* デスクリサーチ：公開情報を収集し整理する
* インタビュー：エキスパートから情報を聴取する
```

## 納品物（共通）

- シナリオ比較表（選択肢A/B/Cの費用対効果）を含める

**【汎用型】の例**

```markdown
## 8. 納品物

〇〇と△△を経営会議で即決できる資料一式として短期間で整える

* **最終報告書（PowerPoint形式）**
  * 現状分析結果（業務フロー・ボトルネック・コスト構造）
  * ベンチマーク比較と推奨アプローチ
  * 選択肢A/B/Cの**シナリオ比較（投資額・効果・リスク・実現期間）**
* **データブック（Excel形式）**
  * 業務量・コスト集計データ
  * ベンチマーク比較一覧
  * ROI試算モデル（前提変数を変更可能）
* **実行ロードマップ**
  * フェーズ別の実行計画（マイルストーン・担当・成果物）
```

**【調査型】の例**

```markdown
## 8. 納品物

〇〇と△△を経営会議で即決できる資料一式として短期間で整える

* **最終報告書（PowerPoint）**
  * 〇〇のレンジ推定（年齢×役割×経験×規模）
  * 競合の〇〇×△△×□□ベンチマーク
  * 選択肢A/B/Cの**シナリオ比較（費用・リスク・獲得確度）**
* **データブック（Excel）**
  * 集計データ（企業/職種/レンジ/要件/訴求）
  * 推定の根拠サマリ（参照ソース、インタビュー件数、集計ロジック）
* **インタビュー議事録**
  * 論点別インサイト（決定要因、辞退要因、交渉余地の実態）
```

## 体制【汎用型】

```markdown
## 9. 体制

貴社側と弊社側の役割を明確に分け、意思決定のスピードを確保する

* **弊社側体制**
  * プロジェクトリード: 1名（全体統括・クライアント窓口）
  * コンサルタント: 1〜2名（分析・設計・資料作成）
  * 専門アドバイザー: 必要に応じて（技術/業界知見）
* **貴社側体制（お願い事項）**
  * プロジェクトオーナー: 1名（最終意思決定権限）
  * 実務担当者: 各部門1名（ヒアリング・データ提供・レビュー）
* **コミュニケーション設計**
  * 週次進捗共有（30分、オンライン）
  * 中間報告会（対面推奨）
  * Slack/Teams等での随時連絡
```

（調査型では「11. 体制・進め方（補足）」として、学習反映の仕組み・リスク管理を簡潔に補足する。汎用型ほど詳細な体制表は必須ではない。）

## スケジュール（共通）

- **週次で分解**する（月単位の大雑把なスケジュールは禁止）
- 各週のアプローチ／タスクと中間共有のタイミングを明確化

```markdown
## スケジュール

設計・分析・検証を週次で回し、中間共有を挟んで〇月末に経営判断用最終報告へ確実に着地する

* **W1（○月上旬〜中旬）**
  * キックオフ、対象範囲確定、情報提供依頼／ベンチマーク企業リスト確定
  * ヒアリング／インタビュースケジュール確定（調査型は対象者母数・代替定義案の確認）
* **W2〜W3（○月中旬〜下旬）**
  * 汎用型：現状分析（業務フロー可視化、システム棚卸し）、ベンチマーク収集開始
  * 調査型：デスクリサーチ、データ収集・構造化、インタビュー対象者のスクリーニング開始
* **W4（○月上旬）**
  * 汎用型：ベンチマーク分析完了
  * 調査型：インタビュー本実施（並行）
  * **中間共有**（暫定結果、論点別の一次示唆、追加深掘り方針）
* **W5〜W6（○月中旬）**
  * 汎用型：シナリオ設計、ROI試算、実行計画策定
  * 調査型：追加インタビュー、推定の精緻化、競合ベンチマーク完成
  * シナリオ比較表のたたき台作成
* **W7〜W8（○月下旬〜納期）**
  * 最終資料化、意思決定比較表／実行ロードマップの確定
  * **納品**
```

## 見積り（共通）

- プランごとに**カバーする論点・アプローチ／タスクの範囲**が異なり、範囲に応じて見積金額が変わる構造にする
- 金額は具体的に記載する（「予算上限内で調整」だけで終わらない）

**【汎用型】**

| プラン | 価格（税抜） | カバー論点 | 含むアプローチ | 想定アウトプット |
| --- | ---: | --- | --- | --- |
| A：ライト | 180万円 | 大論点①②（骨格） | A1: 現状分析＋A2: ベンチマーク | 基本レポート |
| B：スタンダード（推奨） | 280万円 | 大論点①②③ | A＋A3: シナリオ設計＋比較表 | レポート一式＋比較表 |
| C：プレミアム | 380万円 | 全論点（①〜④） | B＋A4: 実行計画＋ロードマップ | 詳細データブック＋実行ロードマップ |

**【調査型】**

| プラン | 価格（税抜） | カバー論点 | 含むタスク | 想定アウトプット |
| --- | ---: | --- | --- | --- |
| A：ライト | 180万円 | 大論点①②（骨格） | T1: デスクリサーチ＋T2: エキスパートINT | 基本レポート |
| B：スタンダード（推奨） | 250万円 | 大論点①②③ | A＋T3: 潜在顧客INT＋シナリオ比較 | レポート一式＋比較表 |
| C：プレミアム | 320万円 | 全論点（①〜④） | B＋T4: セグメント別分析＋T5: 施策設計 | 詳細データブック＋施策ロードマップ |

**母数不足時の方針（調査型は全プラン共通で記載）**：
- 対象を「〇〇」→「△△」へ段階拡張し、〇〇プレミアムを相対比較で示す
- 「決定実績」母数が不足する場合、「提示実績」も併用し推定精度を確保

## 支援実績の伝え方【汎用型】

- 面談前に先方へ共有する**匿名化した実績サマリ**を準備
- 類似案件の経験、アウトプットの型を示し、信頼を構築

```markdown
## 0. 支援実績の伝え方（匿名化）

〇〇と△△を組み合わせ、意思決定資料まで一気通貫で作る

* **先方共有用の実績サマリ文案（固有名詞・数値は伏せる前提）**
  * 〇〇領域で、△△を分析・設計し、経営層決裁用の根拠資料に落とし込む支援を実施
  * 現状分析とベンチマーク比較を突合し、複数シナリオの費用対効果を定量比較する設計を採用
  * 実現リスクに備え、段階的導入計画と代替案を準備
* **アウトプットの型（過去事例で多用する型）**
  * 「現状分析×ベンチマーク×シナリオ比較」の三点セットで、選択肢A/B/Cを比較可能にする
  * フェーズ別の実行計画と、必要投資（工数・費用）を並列表にする
```

## 次アクション【調査型】

```markdown
## 12. 次アクション（3者面談で確認したい事項）

論点優先度と対象定義を揃え最短で調査設計を確定する

* 最重要の意思決定は「〇〇」「△△」「□□」のどれか？
* 優先して深掘りしたいベンチマーク企業群と、特に比較したい企業はどこか？
* 対象定義（役割範囲、期間、地域など）の確定
* 「〇〇値」の取り方：〇〇実例×△△自己申告×□□レンジの突合で、守れる推定ロジックを合意する
* 中間報告で最低限必要なアウトプットは何か？
* 第1フェーズの成功条件：〇月末時点で経営層が「A or B or C」を決裁できる状態をゴールに置く
```
