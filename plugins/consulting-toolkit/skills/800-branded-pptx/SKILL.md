## コンテンツ作成モード

ユーザーがテキストやmdファイルなどのコンテクストを提供した場合、スライド作成に着手する前に、以下の2つのモードのどちらで進めるかをユーザーに確認する。

| モード | 説明 |
|--------|------|
| **忠実再現モード** | 提供されたコンテクストの内容を割愛・省略・文言の変更をせず、そのまま忠実にスライドへ再現する。ただし、1枚のスライドに収まらない場合は、下記「ピラミッド構造」の原則に従って複数スライドに分割することは許容する。 |
| **柔軟作成モード** | 提供されたコンテクストを素材として活用しつつ、スライドとして最適な構成・表現・文言に再編集する。情報の取捨選択、要約、表現の変更を行う。 |

**確認の例**: 「提供いただいた内容をそのまま忠実にスライドに反映しますか？ それとも、スライドに適した形に柔軟に再編集してもよろしいですか？」

※ ユーザーがコンテクストを提供せず、トピックや指示のみを与えた場合はこの確認は不要。

## ピラミッド構造

複数スライドにまたがるパートでは、見出しレベルの異なるスライドを同列に並べない。

**原則**: サマリスライド → 詳細スライド群の順に構成する。
- あるパートを複数枚で説明する場合、まずそのパート全体の要点を概観するサマリスライドを置く
- その後に、個々のトピックを掘り下げる詳細スライドを並べる
- 同一階層のスライド同士でタイトルの粒度を揃える

**例**:
```
[サマリ] 3つの成長戦略（概要を1枚で）
  ├─ [詳細] 戦略①: 新規顧客開拓
  ├─ [詳細] 戦略②: 既存顧客深耕
  └─ [詳細] 戦略③: 新規事業展開
```

## デザイン哲学

**レイアウトと情報設計はコンサルティングレポートを参照する。フレームワーク図・チャート・プロセスフロー・テーブル・アノテーション等を活用し、論点と示唆が直感的にわかるようにする。色・フォント・スライド骨格は本スキルのデザイントークンと骨格ヘルパーに従う。**

### 原則

- **構造の可視化**: スライド上の情報の関係性を、その内容に最も適した表現方法で視覚化する。例: 2列分割、データテーブル、カテゴリ行テーブル、横棒グラフ、プロセスフロー、ツリー/階層図、タイムライン、コールアウト/KPI、セクションサマリ、マルチパネルグリッド。pptxgenjs API で自由に構成する
- **1スライド1メッセージ**: 各スライドに明確なキーメッセージを持たせ、それを支える最適なレイアウトを選ぶ
- **色による意味付け**: アクセントグリーン(#29BA74)とサブブルー(#1789B1)を活用し、異なるカテゴリ・フェーズ・立場を色で区別する
- **レイアウトの多様性**: 意味なく同じレイアウトを連続させない。ただし、同一フレームワークで整理される内容（例：論点①②③をそれぞれ1枚ずつ詳述する等）は、あえて同じレイアウトを揃えることで構造の一貫性を示してよい
- **余白の活用**: 詰め込まない。余白で視覚的な優先度を明確にする
- **クリーンでシンプル**: 装飾要素は最小限に。テキストとレイアウト構造で情報を伝える

### レイアウト選択プロセス

コンテンツスライドを作成する際、まずそのスライドが伝える情報の構造（比較、手順、階層、並列、集約、推移など）を特定し、それに最適なレイアウトを pptxgenjs API で設計する。ボイラープレートの骨格ヘルパー（`addContentSlide` 等）はスライドの枠を提供するが、枠の中のレイアウトは毎回内容に応じて設計する。

連続する2枚以上のコンテンツスライドに同じレイアウトパターンを使う場合、それが意図的な構造の一貫性（同一フレームワークの詳述等）であることを確認する。該当しなければ異なるレイアウトを選択する。

## 禁止パターン（AIっぽさの排除）

以下はAI生成スライドに典型的な「AIっぽさ」の原因であり、本スキルでは**一切使用しない**。

| 禁止事項 | 理由 |
|---------|------|
| **ドロップシャドウ付きカード** | Webコンポーネント由来。スライドでは矩形＋ボーダーまたは背景色で十分 |
| **英語ラベルの見出し** | 「FEATURES」「CASE STUDIES」等。日本語の主張文に置き換える |
| **ネオンカラーのアクセント** | シアン、ティール等のWeb的配色。ブランドカラー（グリーン/ブルー）のみ使用 |
| **スライド内の絵文字** | ビジネス資料では使わない |
| **アイコン＋カードで図解の代替** | 情報の構造を示すべき場面ではフロー図、マトリクス、テーブル等で構造を可視化する |
| **グラデーション背景** | フラットな単色背景のみ使用 |
| **テーブルヘッダーに青/緑背景** | 太字+下線のみでシンプルに区切る |
| **セクションをボックス枠線で囲む** | 見出し+下線またはグレー背景で区切る |
| **ラベル的見出しをアクセント色にする** | 「概要」「ポイント」等はダークグレー(C.text)太字。アクセント色はメッセージ性のある見出しにのみ |
| **レイアウトで区別できる項目を色で塗り分ける** | 位置・ラベル・区切り線で区別できる項目に異なる色を割り当てない。色は「意味の違い」（例：課題 vs 成果、現状 vs 目標）を示す場合にのみ使う |
| **SUB.quaternary（ローズ/ピンク）の安易な使用** | レイアウトやデザインで区別できる場合は使わない。色の違いでしか区別できない場面の最終手段としてのみ |

## 絶対ルール

| ルール | 詳細 |
|--------|------|
| **ロゴ** | ダーク背景スライド: `assets/800-logo.png`（ダーク）を白背景矩形上に配置。コンテンツスライド: ロゴなし |
| **フッター** | 全スライドに `© Eight Hundred, Inc.` を標準搭載 |
| **フォント** | Latin = `Century Gothic`、日本語（EA）= `Meiryo UI`。混在時は `splitTextWithFonts()` を使用 |
| **キーメッセージ** | スライドタイトル直下に配置。そのスライドの結論・要点を完全な文章で記載。カテゴリラベルではなく主張文にする |
| **HEXコード** | `#` を付けない（`"29BA74"` ✓ / `"#29BA74"` ✗） |
| **背景** | コンテンツスライドは白。タイトル/セクション/クロージングのみダークグリーン |
| **ページ番号** | コンテンツスライドの右下に配置 |
| **タイトル** | コンテンツスライドのタイトルは 22pt、**太字** |
| **セパレーター** | タイトル直下にライトグレー薄矩形（0.015"） |
| **テキストカラー** | 本文は `404040`（ダークグレー）。純黒 `000000` は使わない |
| **セクションラベル** | コンテンツスライド上部に小さなグレーテキストでセクション名を表示（`addSectionLabel()`） |
| **フォントサイズ** | カラム数に応じて `fontSize()` で自動決定。手動で固定しない |

## クイックスタート

```
1. consulting-deck-boilerplate.js をプロジェクトディレクトリにコピー
2. assets/800-logo.png も同ディレクトリにコピー
3. Design Tokens（C, SUB, FONT）と骨格ヘルパーはそのまま使う
4. 骨格ヘルパーの後に案件のスライド構築コードを追加
5. node build.js で生成
6. pptxスキルのQA手順でビジュアルQA
```

## 責務の分離

| 本スキルが提供するもの | pptxスキルに委譲するもの |
|---------------------|--------------------------|
| カラーパレット（C, SUB） | pptxgenjs API仕様 |
| フォント設定（FONT） | QA手順（Content QA, Visual QA） |
| スライド骨格ヘルパー（Title, Section, Content, Agenda, Closing, EOF） | 画像変換（soffice, pdftoppm） |
| レイアウトパターン仕様（10パターン） | テキスト抽出（markitdown） |
| ユーティリティ（フォント分割、フォントサイズ、箇条書き） | 依存パッケージのインストール手順 |
| デザイン哲学・禁止パターン | |
| ロゴ画像アセット | |

これにより、別のブランドでスライドを作りたい場合は、本スキルの `C`, `SUB`, `FONT` を差し替えるだけでよい。pptxスキルのメカニクス部分は共通で再利用できる。
