01
Section 01 / Background

企画背景

旅行業界では、商品造成・予約手配・出張手配などの業務において、依然として人手による確認・入力・調整が多く残っている。料金表や条件表はPDF、Excel、FAX、メール添付など多様な形式で流通しており、現場担当者が内容を読み解き、システムへ転記・加工する運用が一般的である。

一方で、近年の生成AI・マルチモーダルAI・AIエージェントの進化により、従来は人間が読み解いていた非構造データをAIが解釈し、業務システムやオペレーションに接続できる可能性が高まっている。本企画では、旅行業界における以下3領域を対象に、AI駆動開発によるPoCテーマを整理する。

  1. ツアー造成システムの設計・進め方のアップデート
  2. パッケージツアー予約体験のアップデート
  3. 出張手配代行（BTM）のアップデート

02
Section 02 / Approach

全体方針

本企画の基本方針は、「既存業務を単純にAIへ置き換える」のではなく、AIを前提に業務プロセス・入力データ・人間のレビュー体験を再設計することである。旅行業界には以下の構造的特徴があり、初期PoCの設計は慎重を要する。

2.1旅行業界の構造的特徴

  • 商品条件・料金条件が複雑で、例外処理が多い
  • 航空会社、ホテル、現地事業者ごとにフォーマットや表現が異なる
  • 業務知識が暗黙知化しており、現場担当者の判断に依存している
  • システム投資余力が限られ、段階的な導入が求められる
  • トップライン向上領域とコスト削減領域で、予算化の難易度が異なる

2.2初期PoCの検証ポイント

初期PoCでは「実運用に耐える完成システム」を目指すのではなく、以下4点を検証する。

  1. AIがどこまで旅行商品の非構造データを読み取れるか
  2. 人間のレビュー・修正を前提にすれば、業務に使える水準まで近づけられるか
  3. 業務プロセスのどこをAI化すれば、費用対効果が出やすいか
  4. 将来的に横展開可能な共通基盤・テンプレート化が可能か
Key Insight
月20〜30万円の予算で「実運用システム」を作るのは現実的ではない。初期投資で買うべきは完成品ではなく、「どこまでできるかを見極める解像度」である。

04
Section 04 / Theme One

テーマ①：ツアー造成システムの設計・進め方のアップデート

4.1現状の課題

現在の旅行商品造成では、航空券、ホテル、現地手配、オプションなどの情報を、担当者が料金表や条件表を見ながら手作業で読み取り、必要な情報をシステムへ入力している。

As-Is / 現行業務
人が読み、人が打ち、人が確かめる
  • 航空券・ホテルなどの料金表を人間が確認
  • PDF / Excel / FAX / メール添付などから必要情報を抽出
  • 商品造成システムへ手入力
  • 不明点や例外条件を現場担当者が判断
  • 商品公開前に人間が確認・修正
To-Be / 目指す姿
AIが読み、人が整え、システムが繋ぐ
  • PDF / ExcelをAIが読み取り
  • 必要項目を構造化データとして抽出
  • 人間がレビュー・修正
  • 修正済みデータをツアー造成に活用
  • 将来的には商品造成・販売システムと接続

4.2PoCで検証したいこと

  1. 旅行商品の料金表・条件表をAIが正しく読み取れるか
  2. 読み取った情報を、ツアー造成に必要な項目へマッピングできるか
  3. 人間がレビュー・修正しやすい形で提示できるか
  4. 限定されたエリア・商品タイプであれば、実務に近い造成フローを再現できるか
  5. 本格開発に向けて、DB設計・レビューUI・運用設計上の論点を洗い出せるか

4.3初期PoCの推奨スコープ

月額20〜30万円程度の予算感では、実運用可能な造成システムの開発まで含めるのは難しい。対象を強く限定した「技術検証＋簡易プロトタイプ」に絞ることが望ましい。

Recommended Scope
限定スコープでの技術検証＋簡易プロトタイプ
  • 対象エリアを1エリアに限定
  • 対象商品タイプを1〜2種類に限定
  • 対象資料をPDF / Excelの代表的な数パターンに限定
  • 抽出項目を最小限に限定（出発日・期間・航空会社・ホテル名・部屋タイプ・料金・適用条件・除外日・注意事項）
  • 人間がレビューする前提で、抽出結果を一覧表示
  • 修正済みデータをCSV / JSONなどで出力

4.4成果物イメージ

最小成果物

  • 対象業務フロー整理、対象資料の分類、AIによる読み取り検証、抽出項目定義
  • 簡易デモ（PDF / Excelアップロード → AIによる項目抽出 → 一覧表示 → 人間による確認・修正 → CSV / JSON出力）
  • 精度・限界・運用上の注意点整理、次フェーズに向けた開発スコープ案

本格開発時に追加検討すべきもの

  • 商品造成DB設計／旅行商品ごとの条件分岐管理／航空券・ホテル・現地手配のマスタ管理
  • 例外条件の管理／承認ワークフロー／既存販売システムとの連携／ログ・監査・差分管理／GDS・API連携

4.5想定リスクと対応方針

リスク内容対応方針
読み取り精度が安定しないPDFやExcelの形式が多様で、AIが誤読する可能性がある対象フォーマットを限定し、人間レビューを前提にする
業界知識が必要料金条件や旅行業特有の例外解釈が必要旅行手配経験者・業務担当者と連携する
DB設計が重い造成システム化には商品構造の設計が必要初期PoCではDB設計を深掘りしすぎない
予算が小さい月20〜30万円では実運用システム開発は困難技術検証・簡易デモ・次フェーズ設計に限定する
トップライン非直結業務効率化テーマのため、投資判断が厳しくなりやすいコスト削減だけでなく、商品造成スピード向上を訴求する

4.6評価

テーマ①は、旅行業界の構造的課題に直結しており、横展開可能性も高い。一方で、業務知識・データ構造・例外処理が重く、初期PoCのスコープを誤ると費用対効果が悪化しやすい。月20〜30万円の予算では「作る」よりも「できる範囲を見極める」位置づけで進めるべきである。

05
Section 05 / Theme Two

テーマ②：パッケージツアー予約体験のアップデート

5.1現状の課題

従来のパッケージツアー販売では、旅行会社が事前に造成したツアーの中から、ユーザーが条件に合う商品を検索・比較・予約する流れが一般的である。しかし、ユーザーの旅行ニーズは本来多様であり、既存商品から探す体験では以下の課題がある。

  • 希望条件に合う商品を探しにくい
  • ユーザーが旅行条件を言語化できない
  • 商品数が多いほど比較負荷が高い
  • 既存ツアーにない柔軟な提案が難しい
  • 相談型の販売体験をオンラインで再現しにくい

5.2目指す姿

ユーザーがAIと対話しながら、希望条件・予算・目的・同行者・制約条件を伝えることで、AIがリアルタイムにツアー案を生成・提案する体験を目指す。

As-Is / 現行体験
商品から検索する
  • 事前造成済みツアーからユーザーが検索
  • 条件に合いそうな商品を比較
  • 不明点は問い合わせ
  • 人間が追加提案・調整
To-Be / 目指す体験
条件から提案される
  • ユーザーがAIに希望を相談
  • AIが条件を整理
  • 候補プランを複数生成
  • 在庫・商品・料金条件を踏まえて提案
  • 必要に応じて担当者が確認・修正
  • 最終的に予約導線へ接続

5.3PoCで検証したいこと

  1. ユーザーの曖昧な旅行相談をAIが整理できるか
  2. 条件に応じた旅行プラン案を生成できるか
  3. 既存商品・料金表・在庫情報を踏まえた提案が可能か
  4. 担当者がレビューできる形で提案内容を出力できるか
  5. CVR向上・問い合わせ増加・商談化など、トップライン寄与の仮説を検証できるか

5.4初期PoCの推奨スコープ

テーマ②はトップライン向上に直結しやすく、役員・部長層への訴求力が高い。一方で、予約完了までAIが自動化するには、在庫、料金、約款、責任範囲、オペレーションの整備が必要である。初期PoCでは予約完了までの完全自動化ではなく、「AI接客・旅行プラン提案・造成案生成」に絞ることが望ましい。

Recommended Scope
AI接客＋プラン提案＋造成補助
  • 対象ユーザー：家族旅行/ハネムーン/シニア旅行/法人/特定方面海外旅行等から選定
  • 対象エリアを限定
  • AIチャットによる希望条件ヒアリング
  • 旅行プラン案の生成
  • 担当者向け造成メモの生成
  • 予約確定は人間が対応
  • 既存商品・料金データとの接続は限定的に実施

5.5成果物イメージ

  • AI旅行相談チャットの簡易デモ／ヒアリング項目設計／ユーザー条件整理ロジック
  • 旅行プラン生成テンプレート／担当者向け造成メモ出力／想定UIワイヤー
  • トップライン効果仮説（問い合わせ率向上・CVR向上・高単価商品の提案率向上・オプション販売率向上）
  • 本格開発に向けた機能ロードマップ

5.6想定リスクと対応方針

リスク内容対応方針
現場反発造成・対話業務に価値を感じる現場が抵抗する可能性AIを代替ではなく補助として位置づける
誤提案リスクAIが存在しない商品や不正確な料金を提案する可能性初期は担当者レビューを必須にする
予約責任の所在AI提案に基づく予約時の責任範囲が曖昧予約確定は人間対応に限定する
データ接続が重い商品・在庫・料金との連携が必要初期はダミーデータまたは限定データで検証する
意思決定の難しさ新しい販売体験のため社内合意形成が必要役員・部長層向けにトップライン効果を訴求する

5.7評価

テーマ②は、売上向上・顧客体験向上・話題性の観点で最も大きな可能性がある。単なる業務効率化ではなく、新しい旅行販売体験の提案として位置づけられるため、予算化の余地も比較的大きい。一方で、現場オペレーションや責任範囲の整理が必要なため、初期PoCでは予約完了までを自動化せず、AIによる相談・提案・造成補助に留めるのが現実的である。

06
Section 06 / Theme Three

テーマ③：出張手配代行（BTM）のアップデート

6.1現状の課題

法人向け出張手配では、チャットやメールで出張条件の相談を受け、人間が航空券・ホテル・移動手段などを調査・提案・手配する運用が一般的である。大型案件を受注した場合、相談件数が急増し、既存の人員では対応が逼迫する可能性がある。

6.2目指す姿

出張相談が来た際に、AIがリアルタイムで条件を整理し、候補案を作成し、担当者に提示する。将来的には、手配依頼や確認連絡の一部まで自動化する。

As-Is / 現行業務
人が解読し、人が手配する
  • 法人顧客からチャットやメールで相談
  • 担当者が条件を整理
  • 航空券・ホテルなどを調査
  • 候補案を作成
  • 顧客へ提案、承認後に手配
To-Be / 目指す業務
AIが整理し、人が判断する
  • 相談内容をAIが自動要約
  • 必要条件をAIが抽出
  • 不足情報をAIが質問
  • 候補案をAIが生成
  • 担当者が確認・修正
  • 顧客返信文や手配依頼文をAIが作成

6.3PoCで検証したいこと

  1. チャット・メールの相談内容から、出張条件を抽出できるか
  2. 不足情報をAIが特定できるか
  3. 担当者向けに候補案・確認事項を整理できるか
  4. 顧客返信文・手配依頼文を生成できるか
  5. 大型案件受注時のオペレーション増加に耐える支援ツールになり得るか

6.4初期PoCの推奨スコープ

テーマ③は、3テーマの中で最も小さく始めやすい。既存業務の相談内容をAIが整理し、担当者を支援する形であれば、実装範囲を比較的限定できる。

Recommended Scope
相談内容の構造化＋担当者支援
  • 対象はチャットまたはメール相談
  • AIが相談内容を要約
  • 出張条件を構造化（出発地・目的地・日程・人数・予算・希望時間帯・宿泊条件・移動制約・承認条件）
  • 不足情報の洗い出し
  • 顧客への確認文生成
  • 担当者向け候補案作成
  • 実予約・決済は人間が対応

6.5成果物イメージ

  • 出張相談AIアシスタントの簡易デモ
  • 相談文からの条件抽出テンプレート／不足情報チェックリスト
  • 顧客返信文生成／担当者向け手配メモ生成
  • 業務削減効果の試算／大型案件受注時の導入ロードマップ

6.6想定リスクと対応方針

リスク内容対応方針
大型案件の受注依存ニーズが案件受注可否に左右される受注前はライトな検証に留める
コスト削減領域の予算化難予算化が難しい可能性がある大型案件対応力・機会損失防止を訴求する
自動手配のリスク誤予約・誤手配の責任が重い初期は人間承認を必須にする
既存ツール連携チャット、メール、予約システムとの接続が必要初期は手動投入・簡易UIで検証する

6.7評価

テーマ③は、実装範囲を限定しやすく、PoCとして最も現実的である。大型案件受注時のオペレーション逼迫という明確な課題があるため、導入理由も説明しやすい。一方で、受注前の段階では予算化が難しい可能性があるため、「大型案件受注時に即PoCへ移行できる準備テーマ」として位置づけるのがよい。

Section 07 / Priority

優先順位案

7.1推奨優先順位

1
First Priority
テーマ② パッケージツアー予約体験

トップライン向上に直結しやすく、役員・部長層への訴求力が高い。単なる業務効率化ではなく、新しい旅行販売体験の提案として位置づけられるため、別予算化の余地も大きい。話題化・事例化のポテンシャルも最も高い。

2
Second Priority
テーマ③ 出張手配代行（BTM）

小さく始めやすく、大型案件受注時のニーズが明確。PoC設計の難度も最も低い。ただし受注前段階での予算化はやや難しいため、「受注時に即PoCへ移行できる準備テーマ」として位置づけるのが現実的。

3
Third Priority
テーマ① ツアー造成システム

業界課題としては最大、ただしスコープ管理が最難。横展開可能性は高いが、20〜30万円ではスコープ管理が難しい。中長期的には最大のリターンが期待できる領域。

8.3初回フェーズの内容

Phase 0

PoC設計・実現性検証

20〜30万円から
期間：2〜4週間

Section 09 / Sales Talk Points

商談時の推奨メッセージ

9.1基本スタンス

  • 3テーマともAI活用テーマとしては魅力がある
  • ただし、20〜30万円で実運用システムを作るのは現実的ではない
  • 初期は「何ができるか」を見極める検証フェーズにするべき
  • トップライン向上につながる②は別予算化を狙う価値が高い
  • ③は大型案件受注時の即応策として小さく始めやすい
  • ①は横展開可能性があるが、スコープ管理が重要

9.2商談で確認すべき事項

全テーマ共通

  • 最も解決したい経営課題は何か／コスト削減と売上向上のどちらを優先するか
  • 予算を持っている部署・意思決定者は誰か／事例化・共同発信は可能か
  • 横展開前提の共同開発に関心があるか

11 / Summary

まとめ：初期は「作る」より、
「できる範囲を見極める」に投資する。

今回の3テーマは、いずれも旅行業界の構造的課題に対してAIを活用する企画として可能性がある。ただし、月20〜30万円の予算で実運用可能なシステム開発まで担うのは難しいため、初期は「技術検証・業務整理・簡易デモ・次フェーズ設計」に絞るべきである。初回商談では開発を請けるのではなく、まず「AI適用可能性検証フェーズ」として提案し、次フェーズで本格PoC・MVP開発へ進める形が望ましい。

01
テーマ② パッケージツアー予約体験

トップライン向上に直結しやすく、予算化・経営層提案に向いている。話題性・事例化のポテンシャルも高い。

02
テーマ③ 出張手配代行（BTM）

小さく始めやすく、大型案件受注時の業務逼迫対策として現実的。受注時の即応テーマとして位置づける。

03
テーマ① ツアー造成システム

業界課題としては大きいが、業務知識・DB設計・例外処理が重く、スコープ管理が重要。中長期で取り組むべき領域。
